> **SQL (Structured Query Language)** — це стандартна мова запитів для взаємодії з реляційними базами даних. Дозволяє створювати, модифікувати та керувати даними через операції вибірки, вставки, оновлення та видалення.

---

> [!abstract] Core Framework 
> **SQL-команди поділяються на чотири категорії за призначенням:**
> 
> - **DDL (Data Definition Language)** — визначення структури: `CREATE`, `ALTER`, `DROP`;
> - **DML (Data Manipulation Language)** — маніпуляція даними: `INSERT`, `SELECT`, `UPDATE`, `DELETE`;
> - **DCL (Data Control Language)** — керування доступом: `GRANT`, `REVOKE`;
> - **TCL (Transaction Control Language)** — керування транзакціями: `COMMIT`, `ROLLBACK`, `SAVEPOINT`.

---

> [!info] Key Insights
> 
> - **DDL — Data Definition Language.** Визначає структуру об'єктів бази даних:
>     - `CREATE` — створення таблиць, індексів, представлень;
>     - `ALTER` — зміна структури існуючих об'єктів (додавання/видалення стовпців, зміна типів даних);
>     - `DROP` — повне видалення об'єктів бази даних.
> - **DML — Data Manipulation Language.** Керує самими даними:
>     - `INSERT` — додавання нових рядків у таблицю;
>     - `SELECT` — вибірка даних з однієї або кількох таблиць;
>     - `UPDATE` — оновлення існуючих даних;
>     - `DELETE` — видалення рядків з таблиці.
> - **DCL — Data Control Language.** Керує правами доступу:
>     - `GRANT` — надання привілеїв користувачам або ролям;
>     - `REVOKE` — позбавлення привілеїв.
> - **TCL — Transaction Control Language.** Керує транзакціями:
>     - `COMMIT` — збереження всіх змін у межах поточної транзакції;
>     - `ROLLBACK` — відміна всіх змін у межах поточної транзакції;
>     - `SAVEPOINT` — встановлення точки збереження для часткового відкату транзакції.

---

> [!example] Практичне застосування

## DDL — визначення структури

### CREATE

Створення нової таблиці.

```sql
CREATE TABLE users (
    id   INT PRIMARY KEY,
    name VARCHAR(100),
    age  INT
);
```

---

### ALTER

Додавання нового стовпця до існуючої таблиці.

```sql
ALTER TABLE users ADD COLUMN email VARCHAR(255);
```

---

### DROP

Повне видалення таблиці.

```sql
DROP TABLE users;
```

---

## DML — маніпуляція даними

### INSERT

Додавання нового рядка у таблицю.

```sql
INSERT INTO users (id, name, age)
VALUES (1, 'John Smith', 30);
```

---

### SELECT

Вибірка всіх даних з таблиці.

```sql
SELECT * FROM users;
```

---

### UPDATE

Оновлення даних у конкретному рядку.

```sql
UPDATE users
SET age = 31
WHERE id = 1;
```

---

### DELETE

Видалення рядка з таблиці.

```sql
DELETE FROM users
WHERE id = 1;
```

---

## DCL — керування доступом

### GRANT

Надання привілею на вибірку даних.

```sql
GRANT SELECT ON users TO readonly_user;
```

---

### REVOKE

Позбавлення привілею.

```sql
REVOKE SELECT ON users FROM readonly_user;
```

---

## TCL — керування транзакціями

### COMMIT / ROLLBACK / SAVEPOINT

```sql
BEGIN;

UPDATE users SET age = 32 WHERE id = 1;

SAVEPOINT before_delete;

DELETE FROM users WHERE id = 2;

/* Якщо щось пішло не так — відкочуємо до точки збереження */
ROLLBACK TO before_delete;

/* Зберігаємо лише UPDATE */
COMMIT;
```

---

> [!tip] Коли використовувати SQL
> 
> **Так** ✅:
> 
> - Потрібна вибірка або зміна даних у реляційній БД;
> - Необхідне управління структурою таблиць (DDL);
> - Потрібен контроль доступу на рівні користувачів (DCL);
> - Критична консистентність даних — використовуйте транзакції (TCL).
> 
> **Ні** ❌:
> 
> - Дані неструктуровані (JSON, зображення, документи) — краще NoSQL;
> - Складні аналітичні агрегації на великих обсягах — потребують оптимізації або спеціалізованих інструментів.

---

> [!warning] Обмеження та ризики
> 
> - **Складні операції.** Агрегація та JOIN між багатьма таблицями вимагають глибокого розуміння SQL і можуть бути неефективними без правильних індексів.
>     
> - **Обмеження реляційної моделі.** Деякі типи даних (зображення, JSON-об'єкти, ієрархічні структури) складно зберігати та ефективно запитувати в реляційній БД.
>     
> - **Небезпека DDL без резервної копії.** `DROP TABLE` або `ALTER TABLE` — незворотні операції; їх виконання без бекапу може призвести до втрати даних.
>     
> - **Відкритий `ROLLBACK`.** Транзакція без `COMMIT` або `ROLLBACK` може утримувати блокування та знижувати продуктивність інших запитів.
>
