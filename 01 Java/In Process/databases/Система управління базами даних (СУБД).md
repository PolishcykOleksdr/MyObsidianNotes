>**СУБД (Система управління базами даних)** — це програмне забезпечення для створення, керування та використання баз даних. Забезпечує ефективний і безпечний доступ до даних, їх збереження, організацію та обробку.

---

> [!abstract] Core Framework
>  **Основні функції СУБД:**
> 
> - **Створення БД** — визначення структури таблиць та взаємозв'язків між ними;
> - **Збереження даних** — надійне і цілісне зберігання на диску або інших носіях;
> - **Запити** — виконання складних операцій отримання, модифікації та видалення даних;
> - **Керування доступом** — захист від несанкціонованого доступу та пошкоджень;
> - **Резервне копіювання та відновлення** — захист від втрати даних у разі аварій;
> - **Підтримка транзакцій** — атомарне виконання групи операцій (або всі успішно, або жодна);
> - **Оптимізація доступу** — швидке виконання запитів навіть при великих обсягах даних.

---

> [!info] Key Insights
> 
> - **Транзакційність (ACID).** СУБД гарантує атомарність, консистентність, ізоляцію та надійність операцій — критично для фінансових та бізнес-систем.
>     
> - **Різноманіття рішень.** Популярні СУБД: MySQL, PostgreSQL, Oracle Database, Microsoft SQL Server, SQLite. Кожна має свої переваги, недоліки та цільову аудиторію.
>     
> - **Відкриті vs. комерційні.** MySQL, PostgreSQL, SQLite — вільні рішення. Oracle Database та Microsoft SQL Server — пропрієтарні з комерційними ліцензіями (мають безкоштовні Express-версії для навчання).
>     
> - **SQLite — особливий випадок.** Легковага вбудована СУБД без окремого сервера; підходить для мобільних додатків та малих проєктів, але має обмежену масштабованість та інтеграцію.
>     
> - **PostgreSQL — де-факто стандарт відкритих СУБД.** Підтримує ACID, повнотекстовий пошук, розподілені запити, горизонтальне та вертикальне масштабування, стандарти SQL та SQL:2011.
>

> - **PostgreSQL** — складна бізнес-логіка, потреба в розширюваності та відповідності стандартам;

> - **Microsoft SQL Server** — інтеграція з екосистемою Microsoft (.NET, Azure);
> - **SQLite** — мобільні додатки, десктопні програми, кешування конфігурацій.

## PostgreSQL — детальний огляд

**PostgreSQL** — потужна відкрита СУБД з акцентом на надійність, розширюваність та відповідність стандартам.

**Ключові особливості:**

- **Відкритий код** — вільна ліцензія, можна використовувати, модифікувати та розповсюджувати безкоштовно;
- **Розширюваність** — модульна архітектура дозволяє підключати додаткові модулі;
- **ACID-транзакції** — атомарність, консистентність, ізоляція та надійність гарантовані на рівні рушія;
- **Масштабованість** — горизонтальне та вертикальне масштабування від малих сайтів до корпоративних систем;
- **Стандарти** — відповідність SQL та SQL:2011, сумісність з існуючими інструментами;
- **Повнотекстовий пошук** — вбудована підтримка індексування та пошуку по тексту;
- **Розподілені обчислення** — підтримка розподілених запитів і таблиць на різних серверах.

---

> [!tip] Коли використовувати СУБД
> 
> **Так** ✅:
> 
> - Великі або зростаючі обсяги структурованих даних;
> - Потрібна організація даних у логічні групи з відносинами між ними;
> - Критична цілісність та консистентність даних;
> - Необхідна підтримка транзакцій для групових операцій.
> 
> **Ні** ❌:
> 
> - Прості, плоскі набори даних без зв'язків (достатньо файлів або key-value сховища);
> - Надвисоке навантаження на запис з мінімальними вимогами до консистентності (краще NoSQL);
> - Обмежені ресурси та бюджет (розглянь SQLite або хмарні managed-рішення).

---

> [!warning] Обмеження та ризики
> 
> - **Вартість.** Впровадження та підтримка комерційних СУБД (Oracle, MS SQL Server) потребують значного бюджету на ліцензії та кваліфікований персонал.
>     
> - **Складність адміністрування.** Налаштування, оптимізація та моніторинг СУБД вимагають спеціалізованих знань — некоректна конфігурація призводить до деградації продуктивності.
>     
> - **Масштабованість реляційних СУБД.** При надвеликих обсягах даних горизонтальне масштабування реляційних СУБД значно складніше, ніж у NoSQL-рішеннях.
>     
> - **PostgreSQL — ресурсомісткість.** Для ефективної роботи з великими обсягами даних потребує значних апаратних ресурсів порівняно з легковаговими альтернативами.
>
