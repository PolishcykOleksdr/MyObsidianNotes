> **Фабричний метод (Factory Method)** — це **породжувальний патерн**, який визначає інтерфейс для створення об'єктів, але залишає підкласам вибір конкретного класу для інстанціювання. Клас делегує процес створення екземплярів своїм підкласам.

---

> [!abstract] Core Framework 
> **Factory Method вирішує задачу створення об'єктів без прив'язки до конкретного класу:**
> 
> - Базовий клас оголошує **фабричний метод**, який повертає об'єкт певного інтерфейсу;
> - **Підкласи** перевизначають цей метод і вирішують, який саме клас інстанціювати;
> - Клієнтський код працює через **абстракцію**, не знаючи про конкретні реалізації.

---

> [!info] Key Insights
> 
> - **Делегування створення.** Базовий клас не знає, який саме об'єкт буде створено — це вирішують підкласи через перевизначення фабричного методу.
>     
> - **Відкритість до розширення.** Щоб додати новий тип об'єкта, достатньо створити новий підклас і перевизначити метод — базовий код не змінюється.
>     
> - **Незалежність від конкретних класів.** Клієнт взаємодіє лише з інтерфейсом `Transport`, не знаючи про `Car` чи `Plane`.
>     

---

> [!example] Практичне застосування
> 
> **Структура патерну:**
> 
> - `TransportCreator` — абстрактний клас із фабричним методом `createTransport()`;
> - `CarCreator`, `PlaneCreator` — конкретні підкласи, що перевизначають фабричний метод;
> - `Transport` — інтерфейс продукту;
> - `Car`, `Plane` — конкретні реалізації продукту.

```java
// Базовий клас для створення транспортного засобу
abstract class TransportCreator {
    // Фабричний метод для створення транспортного засобу
    abstract Transport createTransport();

    // Деякі інші методи, спільні для всіх підкласів
}

// Конкретний підклас, який реалізує фабричний метод
class CarCreator extends TransportCreator {
    // Перевизначення фабричного методу для створення автомобіля
    @Override
    Transport createTransport() {
        return new Car();
    }
}

// Конкретний підклас, який реалізує фабричний метод
class PlaneCreator extends TransportCreator {
    // Перевизначення фабричного методу для створення літака
    @Override
    Transport createTransport() {
        return new Plane();
    }
}

// Базовий клас для всіх видів транспортних засобів
interface Transport {
    void deliver();
}

// Конкретний клас автомобіля
class Car implements Transport {
    @Override
    public void deliver() {
        System.out.println("Доставка автомобілем");
    }
}

// Конкретний клас літака
class Plane implements Transport {
    @Override
    public void deliver() {
        System.out.println("Доставка літаком");
    }
}

public class Main {
    public static void main(String[] args) {
        // Створення автомобіля через фабричний метод CarCreator
        TransportCreator carCreator = new CarCreator();
        Transport car = carCreator.createTransport();
        car.deliver(); // Виведе: Доставка автомобілем

        // Створення літака через фабричний метод PlaneCreator
        TransportCreator planeCreator = new PlaneCreator();
        Transport plane = planeCreator.createTransport();
        plane.deliver(); // Виведе: Доставка літаком
    }
}
```

---

> [!warning] Обмеження та ризики
> 
> - **Розростання підкласів.** Для кожного нового типу продукту потрібен новий підклас-творець, що може призвести до великої ієрархії класів.
>     
> - **Ускладнення для простих випадків.** Якщо логіка створення об'єктів проста і не змінюється — патерн додає зайву складність.
>     

---

> [!tip] Коли використовувати
> 
> **Так** ✅:
> 
> - Заздалегідь невідомо, об'єкти яких класів потрібно створювати
> - Потрібно дати підкласам контроль над процесом створення об'єктів
> - Потрібно розширювати систему новими типами без зміни існуючого коду
> 
> **Ні** ❌:
> 
> - Тип створюваного об'єкта завжди відомий і не змінюється
> - Ієрархія класів вже надто розгалужена
