> **Template Method** — поведінковий патерн, що визначає загальну структуру алгоритму у базовому класі, але дозволяє підкласам перевизначити певні кроки без зміни цієї структури.

---

> [!abstract] Core Framework
>  **Патерн Template Method будується на двох рівнях:**
> 
> - **Abstract class (базовий клас)** — містить шаблонний метод із фіксованою послідовністю кроків та абстрактні методи для кожного кроку.
> - **ConcreteClass (підклас)** — реалізує абстрактні кроки відповідно до свого типу, не змінюючи загальну послідовність.

---

> [!info] Key Insights
> 
> - **Інверсія управління.** Базовий клас викликає методи підкласів, а не навпаки — це так звана техніка «Hollywood Principle»: «не дзвони нам, ми самі подзвонимо».
>     
> - **Фіксований скелет, змінні деталі.** Метод `createDocument()` завжди викликає `initialize()` → `addContent()` → `finalizeDocument()` у цьому порядку. Підкласи не можуть змінити порядок — лише вміст кроків.
>     
> - **Усунення дублювання коду.** Загальна логіка (послідовність кроків) знаходиться в одному місці — базовому класі. Підкласи містять лише те, що унікальне для них.
>     
> - **Розширюваність через наслідування.** Новий тип документу — новий підклас із реалізацією трьох методів. Базовий клас залишається незмінним.
>     

---

> [!example] Практичне застосування
> 
> **Створення електронних документів** — базовий клас `Document` визначає порядок кроків:
> 
> - `initialize()` — підготовка документу (різна для тексту і таблиці)
> - `addContent()` — наповнення специфічним вмістом
> - `finalizeDocument()` — завершення та збереження
> 
> Метод `createDocument()` є незмінним шаблоном — підкласи лише заповнюють «порожнини».

---

## Реалізація

```java
// Абстрактний клас для електронного документу
abstract class Document {
    // Загальний метод створення документу (шаблонний метод)
    public void createDocument() {
        initialize();
        addContent();
        finalizeDocument();
    }

    // Метод для ініціалізації документу
    abstract void initialize();

    // Метод для додавання вмісту документу
    abstract void addContent();

    // Метод для завершення документу
    abstract void finalizeDocument();
}

// Конкретний підклас: Текстовий документ
class TextDocument extends Document {
    @Override
    void initialize() {
        System.out.println("Ініціалізація текстового документу...");
    }

    @Override
    void addContent() {
        System.out.println("Додавання текстового вмісту...");
    }

    @Override
    void finalizeDocument() {
        System.out.println("Завершення текстового документу...");
    }
}

// Конкретний підклас: Таблиця
class Spreadsheet extends Document {
    @Override
    void initialize() {
        System.out.println("Ініціалізація таблиці...");
    }

    @Override
    void addContent() {
        System.out.println("Додавання даних до таблиці...");
    }

    @Override
    void finalizeDocument() {
        System.out.println("Завершення таблиці...");
    }
}

public class Main {
    public static void main(String[] args) {
        Document textDocument = new TextDocument();
        textDocument.createDocument();

        System.out.println();

        Document spreadsheet = new Spreadsheet();
        spreadsheet.createDocument();
    }
}

/* Результат:
   Ініціалізація текстового документу...
   Додавання текстового вмісту...
   Завершення текстового документу...

   Ініціалізація таблиці...
   Додавання даних до таблиці...
   Завершення таблиці...
*/
```

---

> [!warning] Обмеження та ризики
> 
> - **Жорстка прив'язка до наслідування.** Підкласи нерозривно пов'язані з базовим класом. Зміна сигнатури абстрактного методу ламає всі підкласи одночасно.
>     
> - **Порушення LSP при неправильному використанні.** Якщо підклас залишає крок алгоритму порожнім — це порушує принцип підстановки Лісков.
>     
> - **Глибока ієрархія.** При великій кількості варіантів алгоритму ієрархія класів розростається. У таких випадках Strategy (через композицію) є кращою альтернативою.
>     

---

> [!tip] Коли використовувати
> 
> **Так** ✅:
> 
> - Є незмінна послідовність кроків алгоритму з варіативними деталями
> - Потрібно уникнути дублювання спільної логіки між підкласами
> - Підкласи відрізняються лише у конкретних кроках, а не в структурі
> 
> **Ні** ❌:
> 
> - Варіацій алгоритму дуже багато — краще Strategy
> - Підкласи мають кардинально різну загальну структуру
> - Потрібна гнучкість без наслідування — краще композиція
