> **Strategy** — поведінковий патерн, що визначає сімейство алгоритмів, інкапсулює кожен з них та робить їх взаємозамінними. Дозволяє варіювати алгоритми незалежно від клієнтів, що ними користуються.

---

> [!abstract] Core Framework 
> **Патерн Strategy будується на трьох ролях:**
> 
> - **Strategy (інтерфейс)** — визначає спільний контракт для всіх варіантів алгоритму.
> - **ConcreteStrategy** — конкретна реалізація алгоритму (наприклад, `BubbleSortStrategy`, `QuickSortStrategy`).
> - **Context** — зберігає посилання на поточну стратегію та делегує їй виконання через `strategy.sort()`. Стратегію можна замінити у будь-який момент через `setStrategy()`.

---

> [!info] Key Insights
> 
> - **Інкапсуляція алгоритму.** Кожен алгоритм живе у власному класі — зміна або додавання нового алгоритму не торкається ні контексту, ні інших стратегій.
>     
> - **Вибір алгоритму на льоту.** `SortingContext.setStrategy()` дозволяє замінити алгоритм під час виконання без перестворення контексту — це ключова відмінність від простого поліморфізму.
>     
> - **Клієнт не знає деталей.** `SortingContext` знає лише про інтерфейс `SortingStrategy`, а не про конкретні реалізації. Це дотримання принципу залежності від абстракцій (DIP).
>     
> - **Усунення умовних операторів.** Замість `if (type == "bubble") { ... } else if (type == "quick") { ... }` — просто виклик `strategy.sort(array)`.
>     

---

> [!example] Практичне застосування
> 
> **Вибір алгоритму сортування** — контекст делегує сортування поточній стратегії:
> 
> - `BubbleSortStrategy` — $O(n^2)$, підходить для малих масивів
> - `QuickSortStrategy` — $O(n \log n)$ в середньому, ефективний для великих даних
> - Стратегію можна змінити між викликами через `setStrategy()`

---

## Реалізація

```java
// Інтерфейс стратегії
interface SortingStrategy {
    void sort(int[] array);
}

// Конкретна стратегія: сортування методом бульбашки
class BubbleSortStrategy implements SortingStrategy {
    @Override
    public void sort(int[] array) {
        System.out.println("Сортування методом бульбашки");
        // Реалізація сортування бульбашкою
    }
}

// Конкретна стратегія: швидке сортування
class QuickSortStrategy implements SortingStrategy {
    @Override
    public void sort(int[] array) {
        System.out.println("Швидке сортування");
        // Реалізація швидкого сортування
    }
}

// Контекст
class SortingContext {
    private SortingStrategy strategy;

    public SortingContext(SortingStrategy strategy) {
        this.strategy = strategy;
    }

    public void setStrategy(SortingStrategy strategy) {
        this.strategy = strategy;
    }

    public void sortArray(int[] array) {
        strategy.sort(array);
    }
}

public class Main {
    public static void main(String[] args) {
        int[] array = {5, 2, 7, 1, 3};

        SortingContext context = new SortingContext(new BubbleSortStrategy());
        context.sortArray(array); // Сортування методом бульбашки

        context.setStrategy(new QuickSortStrategy());
        context.sortArray(array); // Швидке сортування
    }
}

/* Результат:
   Сортування методом бульбашки
   Швидке сортування
*/
```

---

> [!warning] Обмеження та ризики
> 
> - **Надмірне ускладнення.** Якщо алгоритм один і ніколи не змінюється — патерн лише додає зайву абстракцію та збільшує кількість файлів у проєкті.
>     
> - **Клієнт знає про стратегії.** Хтось повинен вирішити, яку стратегію передати в контекст — ця відповідальність залишається за клієнтським кодом або фабрикою.
>     
> - **Передача даних між стратегією та контекстом.** Якщо стратегії потрібно більше даних ніж передається через метод інтерфейсу — доведеться або розширювати інтерфейс, або передавати весь контекст, що порушує інкапсуляцію.
>     

---

> [!tip] Коли використовувати
> 
> **Так** ✅:
> 
> - Потрібно вибирати між кількома варіантами алгоритму під час виконання
> - Є розростання `if/else` або `switch` по типу алгоритму
> - Алгоритми мають однаковий інтерфейс, але різну реалізацію
> 
> **Ні** ❌:
> 
> - Алгоритм єдиний і ніколи не зміниться
> - Різниця між варіантами мінімальна — достатньо параметра в методі
> - Потрібна висока продуктивність — кожне створення стратегії навантажує heap
