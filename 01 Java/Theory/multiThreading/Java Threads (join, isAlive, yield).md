> Методи `join()`, `isAlive()` та `yield()` дозволяють тонко налаштовувати взаємодію між потоками, контролюючи їх черговість та перевіряючи поточний стан.

---

> [!abstract] Core Framework 
> **Управління станом та синхронізація черговості:**
> 
> - **`join()`**: Змушує поточний потік (наприклад, `main`) зупинити виконання до тих пір, поки потік, на якому викликано метод, не завершить свою роботу ("помре").
>     
> - **`isAlive()`**: Перевіряє, чи було запущено потік і чи він ще не завершився. Повертає `boolean`.
>     
> - **`yield()`**: Підказка планувальнику потоків (Thread Scheduler), що поточний потік готовий поступитися процесорним часом іншим потокам такого ж пріоритету.
>     

---

> [!info] Key Insights 
> - **Гарантована послідовність.** Без `join()` планувальник потоків виконує їх у довільному порядку. Виклик `thread.join()` робить виконання детермінованим: наступний рядок коду не виконається, поки `thread` не фінішує. 
> - **Життєвий цикл.** Метод `isAlive()` враховує перехідний період. Навіть після завершення методу `run()` проходить мікроскопічний проміжок часу, перш ніж потік остаточно змінить стан на "мертвий". 
> - **Поведінка `yield()`.** Це не гарантоване зупинення, а лише "жест доброї волі". Якщо інших потоків з таким самим пріоритетом немає, або всі вони мають нижчий пріоритет, той самий потік може продовжити роботу негайно.

---

> [!example] Практичне застосування
> #### 1. Синхронізація через `join()`
> 
> Метод `join()` дозволяє одному потоку чекати на завершення виконання іншого. Це критично важливо, коли результат роботи другого потоку потрібен першому для подальших обчислень.
> 
> Потік, у якому викликано `t.join()`, переходить у стан очікування до моменту завершення потоку `t`.
> 
> ```java
> Thread t = new Thread(myRunnable);
> t.start();
> 
> // Основний потік чекатиме тут, доки потік 't' не закінчить роботу
> t.join(); 
> // Якийсь код....
> ```
> 
> #### 2. Перевірка стану через `isAlive()`
> 
> Цей метод повертає `boolean` і дозволяє дізнатися, чи потік уже був запущений і чи він ще не завершив свою роботу (тобто перебуває в стані виконання або очікування).
> 
> - **`true`**: потік активний.
>     
> - **`false`**: потік ще не запущений або вже завершений.
>     
> ```java
> if (thread.isAlive()) {
>     System.out.println("Потік усе ще працює...");
> }
> ```
> 
> #### 3. Використання `yield()` для передачі пріоритету
> 
> Метод `yield()` — це "підказка" для **Thread Scheduler**, що поточний потік готовий поступитися своїм часом на процесорі іншим потокам такого ж пріоритету.
> 
> - **Механіка:** Потік переходить зі стану _Running_ у стан _Runnable_.
>     
> - **Особливість:** Планувальник може проігнорувати цю підказку, тому `yield()` не гарантує негайної зупинки потоку.
> 
> ```java
> public void run() {
>     for (int i = 0; i < 5; i++) {
>         // Потік сигналізує, що готовий пропустити інших
>         Thread.yield(); 
>         System.out.println(Thread.currentThread().getName() + " виконується");
>     }
> }
> ```

### Резюме методів

| **Метод**       | **Дія**    | **Результат**                             |
| --------------- | ---------- | ----------------------------------------- |
| **`join()`**    | Очікування | Гарантує послідовність виконання потоків  |
| **`isAlive()`** | Перевірка  | Дозволяє моніторити статус об'єкта потоку |
| **`yield()`**   | Поступка   | Спроба оптимізувати розподіл ресурсів ЦП  |

---

> [!warning] Важливо знати 
> - **InterruptedException.** Метод `join()` (як і `sleep()`) викидає це перевіряєме виключення, яке необхідно обробляти через `try-catch` або `throws`. 
> - **Ризик `yield()`.** Не варто покладатися на цей метод для критичної логіки синхронізації, оскільки його поведінка повністю залежить від планувальника JVM і конкретної ОС. 
> - **Термінальні стани.** Як тільки `isAlive()` повернув `false` після завершення роботи, запустити цей самий об'єкт потоку знову неможливо.

---

> [!tip] Коли використовувати 
> - **join()**: Коли результат роботи одного потоку критично необхідний для початку роботи іншого. 
> - **yield()**: Для покращення розподілу ресурсів у системах, де багато однотипних обчислювальних потоків, щоб уникнути "голодування" (starvation) інших потоків.
