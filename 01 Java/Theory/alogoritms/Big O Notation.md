> Ефективність будь-якого алгоритму оцінюється за двома ключовими аспектами: часовою складністю (кількість операцій) та просторовою складністю (обсяг споживаної пам'яті) залежно від розміру вхідних даних $n$.

---

> [!abstract] Core Framework
> 
> **Core Framework**
> 
> Оцінка алгоритмів базується на аналізі споживання ресурсів системи:
> 
> - **Часова складність**: показник часу або кількості ітерацій, необхідних для завершення задачі.
>     
> - **Просторова складність**: обсяг RAM, необхідний для зберігання вхідних даних та створення допоміжних структур.
>     
> - **Верхня межа**: формальне визначення максимального споживання ресурсів при зростанні $n$.
>     

---

> [!info] Key Insights
> 
> - **Асимптотичний аналіз**. Нотація "Велике O" ($O$) описує граничну поведінку алгоритму, ігноруючи константи та менші доданки.
>     
> - **Масштабованість**. Вибір алгоритму з кращим Big O (наприклад, $O(\log n)$ замість $O(n)$) стає критичним лише при значних обсягах даних.
>     
> - **Ресурсний компроміс (Trade-off)**. Часто пришвидшення виконання (зменшення часової складності) вимагає виділення додаткової пам'яті (збільшення просторової складності).
>     
> - **Залежність від середовища**. Оптимальність алгоритму визначається не лише формулою, а й обмеженнями цільового пристрою (наприклад, IoT vs Server).
>

> [!example] Приклади складності в Java
> 
> **1. Константний час $O(1)$**
> 
> Доступ до елемента масиву за індексом.
> 
> Java
> 
> ```java
> /* Час виконання не залежить від розміру масиву */
> int value = array[5]; 
> ```
> 
> **2. Лінійний час $O(n)$**
> 
> Простий перебір елементів у циклі.
> 
> Java
> 
> ```java
> /* Час зростає прямо пропорційно кількості елементів n */
> for (int i = 0; i < n; i++) {
>     System.out.println(array[i]);
> }
> ```
> 
> **3. Квадратичний час $O(n^2)$**
> 
> Вкладені цикли (наприклад, у простих сортуваннях).
> 
> Java
> 
> ```java
> /* Кількість операцій зростає як квадрат від n */
> for (int i = 0; i < n; i++) {
>     for (int j = 0; j < n; j++) {
>         /* Логіка порівняння */
>     }
> }
> ```

---

> [!warning] Обмеження та ризики
> 
> - **Експоненціальний вибух $O(2^n)$**. Алгоритми з такою складністю стають неможливими для виконання навіть при невеликих значеннях $n$.
>     
> - **Пам'ять vs Швидкість**. Швидкий алгоритм може спричинити `OutOfMemoryError`, якщо його просторова складність занадто висока для конкретної JVM.
>     
> - **Константи мають значення**. На малих наборах даних алгоритм з $O(n^2)$, але малою константою, може працювати швидше, ніж $O(n \log n)$ з великими накладними витратами.
>     

---

> [!tip] Коли використовувати
> 
> **Оптимізація за часом**
> 
> - Так ✅: Коли критична швидкість відгуку (Real-time системи).
>     
> - Ні ❌: Якщо це призводить до критичного дефіциту оперативної пам'яті.
>     
> 
> **Оптимізація за пам'яттю**
> 
> - Так ✅: Пристрої з обмеженими ресурсами (Embedded, Mobile).
>     
> - Ні ❌: Якщо це сповільнює роботу системи до неприйнятного рівня.
>

# Порівняння рівнів складності (Big O)

|**Нотація**|**Назва**|**Характеристика зростання**|**Приклад**|
|---|---|---|---|
|$O(1)$|Константний|Не залежить від даних|Доступ за індексом|
|$O(\log n)$|Логарифмічний|Зростає повільно|Бінарний пошук|
|$O(n)$|Лінійний|Прямо пропорційно|Пошук у несортованому списку|
|$O(n \log n)$|Лінійно-логарифмічний|Ефективне зростання|Merge Sort / Quick Sort|
|$O(n^2)$|Квадратичний|Квадратичне зростання|Bubble Sort|
|$O(2^n)$|Експоненціальний|Надзвичайно швидке|Рекурсивне число Фібоначчі|
