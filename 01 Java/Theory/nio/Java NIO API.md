> **Java NIO (New Input/Output)** — це альтернатива стандартному пакету IO, представлена в Java 1.4 для забезпечення високої продуктивності та інтенсивнішої роботи з даними. На відміну від класичного IO, який орієнтований на потоки (stream-oriented), NIO базується на **буферах та каналах** (buffer-oriented).

---

> [!abstract] Core Framework
> 
> Архітектура Java NIO тримається на трьох "китах":
> 
> - **Канали (Channels)**: Шлюзи для передачі даних (аналоги потоків, але двонаправлені).
>     
> - **Буфери (Buffers)**: Контейнери для даних. Весь обмін у NIO відбувається виключно через них.
>     
> - **Селектори (Selectors)**: Механізм, що дозволяє одному потоку (thread) обслуговувати кілька каналів одночасно.
>     

---

> [!info] Key Insights
> 
> - **Буферна орієнтованість**: Ви не читаєте дані безпосередньо з каналу в програму. Ви читаєте їх у буфер, а потім працюєте з ними. Це дозволяє довільно переміщатися по даних (вперед/назад).
>     
> - **Двонаправленість**: Канал може бути відкритий одночасно і на читання, і на запис.
>     
> - **Неблокуючий ввід-вивід (Non-blocking IO)**: Потік може запросити читання даних із каналу і, якщо даних наразі немає, не "зависнути", а продовжити виконання інших завдань.
>     
> - **Мультиплексування**: За допомогою селекторів один потік може ефективно керувати тисячами з'єднань.
>

### Основні компоненти NIO

|**Компонент**|**Опис**|
|---|---|
|**Buffer**|Область пам'яті (наприклад, `ByteBuffer`, `CharBuffer`), де дані тимчасово зберігаються. Має властивості: `capacity`, `position`, `limit`.|
|**Channel**|Представляє з’єднання з файлом, сокетом або іншим ресурсом. Приклади: `FileChannel`, `SocketChannel`.|
|**Charset**|Декодер/енкодер, який перетворює послідовність байтів у символи Unicode та навпаки.|
|**Selector**|Об'єкт, який моніторить кілька каналів на наявність подій (наприклад, "дані готові до читання").|

---

> [!warning] Складність обробки
> 
> Оскільки NIO надає "сирі" дані в буфері, розробнику необхідно самостійно:
> 
> 1. Стежити за тим, чи всі дані завантажилися для повної обробки.
>     
> 2. Використовувати методи `flip()`, `clear()` або `compact()` для коректного перемикання буфера між режимами читання та запису.
>     
> 3. Уникати перезапису даних, які ще не були оброблені програмою.
>     

---

> [!tip] Коли використовувати
> 
> Так ✅:
> 
> - Створення високонавантажених серверів (наприклад, чат-сервери або проксі).
>     
> - Обробка великої кількості одночасних мережевих з'єднань.
>     
> - Робота з великими файлами, де потрібен довільний доступ до даних.
>     
> 
> Ні ❌:
> 
> - Прості послідовні операції з невеликими файлами (класичний IO простіший у реалізації).
>     
> - Коли продуктивність не є критичним фактором.
>
