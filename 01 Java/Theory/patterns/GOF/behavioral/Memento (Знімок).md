> **Memento** — поведінковий патерн, що дозволяє зберігати та відновлювати попередній стан об'єкта без розкриття деталей його внутрішньої реалізації.

---

> [!abstract] Core Framework 
> **Патерн Memento будується на трьох ролях:**
> 
> - **Originator (Редактор)** — об'єкт, стан якого потрібно зберігати. Сам створює знімок через `createMemento()` та відновлює себе через `restore()`.
> - **Memento (Знімок)** — незмінний об'єкт, що зберігає копію стану `Originator`. Не надає зовнішньому коду способу змінити збережені дані.
> - **Caretaker (Охоронець)** — зберігає знімки (у прикладі — змінна `memento` у `main`), але не заглядає всередину і не змінює їх.

---

> [!info] Key Insights
> 
> - **Інкапсуляція стану.** `EditorMemento` оголошує поле `content` як `final` і надає лише геттер. Зовнішній код не може змінити збережений стан — лише прочитати через `getContent()`.
>     
> - **Originator керує власним знімком.** Лише `Editor` знає, як створити знімок (`createMemento`) і як відновитися з нього (`restore`). Логіка серіалізації стану не виходить назовні.
>     
> - **Основа для Undo/Redo.** Якщо зберігати знімки у стеку (`Deque<EditorMemento>`), отримуємо повноцінну багаторівневу відміну дій — стандартний паттерн для текстових редакторів та графічних застосунків.
>     
> - **Snapshot у часі.** Знімок є незалежною копією стану на конкретний момент — подальші зміни `Editor` не впливають на вже збережений `EditorMemento`.
>     

---

> [!example] Практичне застосування
> 
> **Текстовий редактор з відміною дій:**
> 
> 1. Встановлюємо контент «Перший варіант»
> 2. Зберігаємо знімок через `createMemento()`
> 3. Змінюємо контент на «Другий варіант»
> 4. Відновлюємо попередній стан через `restore(memento)`
> 5. Редактор повертається до «Перший варіант»

---

## Реалізація

```java
// Клас, який представляє стан редактора
class EditorMemento {
    private final String content;

    public EditorMemento(String content) {
        this.content = content;
    }

    public String getContent() {
        return content;
    }
}

// Клас, який представляє редактор
class Editor {
    private String content;

    public void setContent(String content) {
        this.content = content;
    }

    public String getContent() {
        return content;
    }

    // Метод для створення знімка поточного стану редактора
    public EditorMemento createMemento() {
        return new EditorMemento(content);
    }

    // Метод для відновлення редактора до попереднього стану
    public void restore(EditorMemento memento) {
        content = memento.getContent();
    }
}

public class Main {
    public static void main(String[] args) {
        Editor editor = new Editor();

        // Редагування контенту
        editor.setContent("Перший варіант контенту");

        // Збереження поточного стану редактора
        EditorMemento memento = editor.createMemento();

        // Редагування контенту
        editor.setContent("Другий варіант контенту");

        // Відновлення редактора до попереднього стану
        editor.restore(memento);

        // Виведення контенту
        System.out.println("Поточний стан редактора: " + editor.getContent());
    }
}

/* Результат:
   Поточний стан редактора: Перший варіант контенту
*/
```

---

> [!warning] Обмеження та ризики
> 
> - **Витрати пам'яті.** Кожен знімок — це копія стану. Для великих об'єктів або частого збереження це може призвести до значного споживання пам'яті. Необхідно обмежувати глибину історії.
>     
> - **Поверхневе копіювання.** Якщо стан містить посилання на мутабельні об'єкти (наприклад, `List`), просте присвоєння у конструкторі `EditorMemento` збереже посилання, а не копію — зміни зовні зіпсують знімок. Потрібне глибоке копіювання.
>     
> - **Мова не захищає від порушення інкапсуляції.** У Java немає вбудованого механізму «friend class» — якщо `getContent()` у `EditorMemento` публічний, будь-який код може прочитати збережений стан. Для строгої інкапсуляції використовують вкладені класи.
>     

---

> [!tip] Коли використовувати
> 
> **Так** ✅:
> 
> - Потрібна функція «Скасувати дію» (Undo/Redo)
> - Потрібно зберігати контрольні точки стану об'єкта
> - Пряме отримання стану порушувало б інкапсуляцію
> 
> **Ні** ❌:
> 
> - Стан об'єкта дуже великий — знімки займатимуть надто багато пам'яті
> - Стан змінюється дуже часто — зберігання кожної зміни нераціональне
> - Достатньо простого поля для збереження попереднього значення
