> **GRASP (ч.3)** завершує розгляд принципів трьома патернами: **Pure Fabrication**, **Indirection** та **Protected Variations** — усі три спрямовані на захист системи від змін та зниження залежностей між компонентами.

---

> [!abstract] Core Framework 
> **Три принципи третьої частини GRASP:**
> 
> - **Pure Fabrication** — створення штучних класів, що не мають відповідника в доменній моделі, але покращують модульність і розподіл відповідальностей;
> - **Indirection** — введення проміжного шару (інтерфейсу, проксі) між класами для зменшення прямих залежностей;
> - **Protected Variations** — захист внутрішнього коду від змін у зовнішніх компонентах через абстракцію.

---

> [!info] Key Insights
> 
> - **Pure Fabrication — це «службовий» клас.** Він не моделює реальний об'єкт предметної області, але бере на себе технічну відповідальність (робота з БД, мережею, файловою системою), яку недоречно класти на доменні класи.
>     
> - **Indirection = залежність від абстракції.** Клієнт взаємодіє не з конкретною реалізацією, а з інтерфейсом. Заміна реалізації відбувається «за лаштунками» — клієнт нічого не знає і не змінюється.
>     
> - **Protected Variations — принцип «точок стабільності».** Визначай, що буде змінюватись (джерела даних, зовнішні API), і огортай це абстракцією. Решта системи стає ізольованою від цих змін.
>     
> - **Зв'язок між принципами.** Indirection і Protected Variations — це фактично два погляди на одну ідею: перший описує механізм (проміжний шар), другий — мотивацію (захист від змін). Pure Fabrication часто є реалізаційним інструментом для обох.
>     
> - **Тестованість.** Усі три принципи напряму покращують тестованість: штучні класи мокуються, інтерфейси підміняються стабами, а захищені варіації дозволяють тестувати бізнес-логіку незалежно від інфраструктури.
>

## 7. Pure Fabrication (Чиста вигадка)

Створюйте класи, які **не є частиною доменної моделі**, але потрібні для покращення модульності та зручності використання системи — наприклад, для взаємодії з БД, мережею або зовнішніми сервісами.

> [!example] Практичне застосування
> 
> `DataProcessor` — це **чиста вигадка**: він не відповідає жодній реальній сутності предметної області, але бере на себе технічну відповідальність за обробку та делегування збереження даних.
> 
> ```java
> /* Клас для роботи з базою даних.
>    Це також Pure Fabrication — технічний клас,
>    що ізолює логіку збереження від решти системи. */
> public class DatabaseService {
>     /* Метод для збереження даних у базі даних.
>        Object — спрощений тип для демонстрації принципу. */
>     public void saveData(Object data) {
>         /* У реальному коді тут був би SQL-запит або ORM-виклик */
>         System.out.println("Дані успішно збережено у базі даних.");
>     }
> }
> 
> /* Клас-чиста вигадка для обробки даних.
>    DataProcessor не є частиною доменної моделі —
>    він існує лише для розділення логіки обробки
>    і логіки збереження даних. */
> public class DataProcessor {
>     /* Посилання на сервіс бази даних.
>        Ін'єкція через конструктор — легко підмінити у тестах. */
>     private DatabaseService databaseService;
> 
>     /* Конструктор, що ініціалізує посилання на сервіс бази даних */
>     public DataProcessor(DatabaseService databaseService) {
>         this.databaseService = databaseService;
>     }
> 
>     /* Метод для обробки та збереження даних.
>        Два кроки: спочатку обробка (тут — логування),
>        потім делегування збереження до DatabaseService. */
>     public void processData(Object data) {
>         /* Логіка обробки даних */
>         System.out.println("Дані успішно оброблено.");
> 
>         /* Збереження оброблених даних у базі даних.
>            DataProcessor не знає деталей збереження — делегує. */
>         databaseService.saveData(data);
>     }
> }
> 
> public class Main {
>     public static void main(String[] args) {
>         /* Створення сервісу бази даних */
>         DatabaseService databaseService = new DatabaseService();
> 
>         /* Створення об'єкта-чистої вигадки для обробки даних.
>            databaseService передається через конструктор (DI). */
>         DataProcessor dataProcessor = new DataProcessor(databaseService);
> 
>         /* Створення даних для обробки */
>         Object data = new Object();
> 
>         /* Обробка та збереження даних */
>         dataProcessor.processData(data);
> 
>         /* Результат:
>            Дані успішно оброблено.
>            Дані успішно збережено у базі даних. */
>     }
> }
> ```

## 8. Indirection (Перенаправлення)

Використовуйте **проміжні об'єкти або шари** для зменшення прямих зв'язків між класами. Введення інтерфейсу або проксі дозволяє змінювати реалізацію без зміни клієнтського коду.

> [!example] Практичне застосування
> 
> `Client` взаємодіє виключно з інтерфейсом `Service`. Конкретна реалізація `ConcreteService` може бути замінена будь-якою іншою — `Client` про це нічого не знатиме.
> 
> ```java
> /* Інтерфейс — точка перенаправлення.
>    Client залежить лише від цього контракту,
>    а не від конкретної реалізації. */
> interface Service {
>     void performAction();
> }
> 
> /* Конкретна реалізація сервісу.
>    Якщо з'явиться інша реалізація — Client міняти не потрібно. */
> class ConcreteService implements Service {
>     @Override
>     public void performAction() {
>         System.out.println("Виконання дії в конкретному сервісі.");
>     }
> }
> 
> /* Клас, що використовує сервіс через інтерфейс.
>    Indirection реалізовано: Client → Service (інтерфейс) → ConcreteService */
> class Client {
>     /* final — після ін'єкції сервіс не підміняється */
>     private final Service service;
> 
>     /* Конструктор, що отримує інтерфейс сервісу через параметр.
>        Конкретна реалізація вирішується зовні — не тут. */
>     public Client(Service service) {
>         this.service = service;
>     }
> 
>     /* Метод для виклику дії сервісу через інтерфейс.
>        Client не знає, що саме відбувається всередині performAction(). */
>     public void doAction() {
>         System.out.println("Виклик дії через інтерфейс.");
>         service.performAction();
>     }
> }
> 
> public class Main {
>     public static void main(String[] args) {
>         /* Створення конкретного сервісу */
>         ConcreteService concreteService = new ConcreteService();
> 
>         /* Client отримує реалізацію через інтерфейс.
>            Підмінити ConcreteService на іншу реалізацію —
>            лише тут, в одному рядку, без торкання Client. */
>         Client client = new Client(concreteService);
> 
>         /* Виклик дії клієнтом через інтерфейс */
>         client.doAction();
> 
>         /* Результат:
>            Виклик дії через інтерфейс.
>            Виконання дії в конкретному сервісі. */
>     }
> }
> ```

## 9. Protected Variations (Захист змін)

Розділяйте систему на компоненти так, щоб **зміни в одній частині не впливали на інші**. Створюй абстракцію навколо того, що може змінитись — і внутрішній код залишиться стабільним.

> [!example] Практичне застосування
> 
> `DataService` захищений від змін у джерелах даних через інтерфейс `DataSource`. Перехід з бази даних на зовнішній API — лише одна зміна ззовні, `DataService` залишається незмінним.
> 
> ```java
> /* Інтерфейс-абстракція для різних джерел даних.
>    Це «точка захисту» — DataService ніколи не побачить
>    конкретні реалізації, лише цей контракт. */
> interface DataSource {
>     void fetchData();
> }
> 
> /* Конкретна реалізація джерела даних — база даних */
> class DatabaseDataSource implements DataSource {
>     @Override
>     public void fetchData() {
>         System.out.println("Завантаження даних з бази даних...");
>         /* У реальному коді тут був би JDBC або ORM-виклик */
>     }
> }
> 
> /* Конкретна реалізація джерела даних — зовнішній API */
> class ApiDataSource implements DataSource {
>     @Override
>     public void fetchData() {
>         System.out.println("Запит на зовнішній API для отримання даних...");
>         /* У реальному коді тут був би HTTP-запит */
>     }
> }
> 
> /* Сервіс, який використовує джерело даних через абстракцію.
>    DataService захищений від змін у реалізаціях DataSource. */
> class DataService {
>     /* final — після ін'єкції джерело не підміняється у рамках одного об'єкта */
>     private final DataSource dataSource;
> 
>     /* Конструктор, який приймає об'єкт абстракції джерела даних.
>        Конкретна реалізація визначається ззовні — Dependency Injection. */
>     public DataService(DataSource dataSource) {
>         this.dataSource = dataSource;
>     }
> 
>     /* Метод для отримання даних.
>        DataService делегує роботу dataSource —
>        він не знає і не хоче знати, звідки беруться дані. */
>     public void fetchData() {
>         dataSource.fetchData();
>     }
> }
> 
> public class Main {
>     public static void main(String[] args) {
>         /* Створення об'єкта бази даних як джерела даних */
>         DataSource database = new DatabaseDataSource();
> 
>         /* Створення сервісу, який використовує базу даних */
>         DataService databaseService = new DataService(database);
> 
>         /* Отримання даних через сервіс */
>         databaseService.fetchData();
> 
>         /* Тепер змінюємо джерело даних на зовнішній API.
>            DataService не змінився — лише інша реалізація DataSource. */
>         DataSource api = new ApiDataSource();
>         databaseService = new DataService(api);
> 
>         /* Знову отримуємо дані, але тепер через зовнішній API */
>         databaseService.fetchData();
> 
>         /* Результат:
>            Завантаження даних з бази даних...
>            Запит на зовнішній API для отримання даних... */
>     }
> }
> ```

---

> [!warning] Обмеження та ризики
> 
> - **Pure Fabrication може стати «сміттєвим класом».** Якщо кидати в штучний клас будь-яку незручну логіку — він швидко перетвориться на клас з низькою когерентністю. Кожна «вигадка» повинна мати чітку єдину відповідальність.
>     
> - **Indirection додає шари абстракції.** Надмірне введення інтерфейсів і проксі ускладнює читання коду та відлагодження — стек-трейс стає глибшим, а потік виконання — менш очевидним.
>     
> - **Protected Variations вимагає прогнозування змін.** Якщо захищати все підряд — архітектура стає надмірно складною. Захист виправданий лише там, де зміна дійсно ймовірна (зовнішні API, формати даних, провайдери).
>     
> - **Усі три принципи — не срібна куля.** Вони вирішують проблему стабільності архітектури, але не замінюють правильного доменного моделювання та інших практик (SOLID, GoF-патерни).
>     

---

> [!tip] Коли використовувати
> 
> **Pure Fabrication** ✅:
> 
> - Технічна відповідальність (БД, мережа, кеш) не личить жодному доменному класу
> - Потрібно ізолювати інфраструктурний код від бізнес-логіки
> - Хочеш покращити тестованість через мокування технічного шару
> 
> **Indirection** ✅:
> 
> - Є пряма залежність між двома класами, яку потрібно розірвати
> - Реалізація може змінитись або підмінятись у тестах
> - Потрібна можливість додати поведінку (логування, кеш) між викликом і реалізацією
> 
> **Protected Variations** ✅:
> 
> - Є зовнішній компонент (API, БД, файлова система), що може змінитись
> - Різні середовища (dev/prod) потребують різних реалізацій
> - Хочеш убезпечити бізнес-логіку від «хвиль змін» в інфраструктурі
