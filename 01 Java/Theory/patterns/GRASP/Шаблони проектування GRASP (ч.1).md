> **GRASP (General Responsibility Assignment Software Patterns)** — це набір принципів та патернів проектування, що допомагають визначати, **які класи та об'єкти** повинні відповідати за певні функції в системі, а також **як вони взаємодіють** між собою.

---

> [!abstract] Core Framework 
> **Основна ідея GRASP** — призначити обов'язки об'єктам так, щоб система була максимально:
> 
> - **незалежною** — об'єкти слабко зв'язані між собою;
> - **повторюваною** — компоненти можна переносити між проектами;
> - **масштабованою** — систему легко розширювати без переписування.
> 
> Розглянуті принципи:
> 
> - **Expert** — призначай обов'язок тому, хто має потрібні дані;
> - **Creator** — створення об'єктів покладай на того, хто знає що й навіщо створювати;
> - **Controller** — управління потоком взаємодії з користувачем — окремий клас.

---

> [!info] Key Insights
> 
> - **Розподіл відповідальності.** Кожен клас займається тільки тим, до чого він «компетентний» — або він зберігає дані, або обробляє запити, або створює об'єкти.
>     
> - **Низьке зчеплення (Low Coupling).** Класи слабко залежать одне від одного, тому зміна одного не «ламає» решту.
>     
> - **Висока когерентність (High Cohesion).** Кожен клас зосереджений на одній чіткій ролі, а не робить «все й одразу».
>     
> - **Адаптивність.** GRASP можна застосовувати на будь-якому етапі — від аналізу вимог до тестування.
>     
> - **Досвід потрібен.** Неправильне застосування GRASP погіршує архітектуру, а не покращує — принципи суб'єктивні та вимагають розуміння контексту.
>     

---

> [!tip] Переваги / Недоліки
> 
> **Так** ✅:
> 
> - Прості та зрозумілі принципи — легко пояснити команді
> - Сприяє низькому зчепленню та високій когерентності
> - Застосовується на різних етапах проектування
> - Вбирає в себе найкращі практики проектування ПЗ
> 
> **Ні** ❌:
> 
> - Не є універсальним — деякі ситуації потребують інших підходів
> - Деякі принципи суб'єктивні та залежать від інтерпретації
> - Може бути непридатним для специфічних архітектур
> - Вимагає досвіду — без нього легко застосувати невірно

## 1. Expert (Експерт)

Призначай обов'язок тому класу, який має **найбільше інформації**, потрібної для його виконання.

> [!example] Практичне застосування
> 
> Клас `Student` зберігає дані про студента — він і є **експертом** щодо цих даних. Клас `StudentService` використовує ці дані через публічні методи, не порушуючи інкапсуляцію.
> 
> ```java
> // Клас, що містить інформацію про студента
> class Student {
>     private String name;
>     private int age;
> 
>     /* Конструктор — ініціалізує поля студента */
>     public Student(String name, int age) {
>         this.name = name;
>         this.age = age;
>     }
> 
>     /* Методи доступу до даних — єдина точка читання полів */
>     public String getName() {
>         return name;
>     }
> 
>     public int getAge() {
>         return age;
>     }
> }
> 
> // Клас, який використовує інформацію про студента
> class StudentService {
>     /* StudentService НЕ зберігає дані про студента —
>        він лише отримує їх через публічний API класу Student.
>        Це і є принцип Expert: обов'язок надавати дані
>        залишається у того, хто їх «знає». */
>     public void printStudentInfo(Student student) {
>         System.out.println("Ім'я студента: " + student.getName());
>         System.out.println("Вік студента: " + student.getAge());
>     }
> }
> 
> public class Main {
>     public static void main(String[] args) {
>         /* Створення об'єкту студента */
>         Student student = new Student("Іван", 20);
> 
>         /* StudentService отримує об'єкт ззовні — він не знає,
>            як створюється Student, але знає, як з ним працювати */
>         StudentService studentService = new StudentService();
>         studentService.printStudentInfo(student);
> 
>         /* Результат:
>            Ім'я студента: Іван
>            Вік студента: 20 */
>     }
> }
> ```

## 2. Creator (Творець)

Покладай відповідальність за **створення об'єктів** на той клас, що має найбільше інформації про те, _що_ і _з якими параметрами_ створювати.

> [!example] Практичне застосування
> 
> Клас `ProductFactory` бере на себе роль **творця** — він знає, як правильно сконструювати `Product`. Клієнтський код більше не відповідає за деталі створення.
> 
> ```java
> // Клас, що представляє товар
> class Product {
>     private String name;
>     private double price;
> 
>     /* Конструктор із параметрами — прийматиме дані від фабрики */
>     public Product(String name, double price) {
>         this.name = name;
>         this.price = price;
>     }
> 
>     public String getName() { return name; }
>     public double getPrice() { return price; }
> }
> 
> // Клас-творець: ізолює логіку створення об'єктів
> class ProductFactory {
>     /* static — можна викликати без інстанціювання фабрики.
>        Метод повертає готовий Product, вже налаштований.
>        Клієнтський код не торкається new Product(...) напряму. */
>     public static Product createProduct(String name, double price) {
>         return new Product(name, price);
>     }
> }
> 
> public class Main {
>     public static void main(String[] args) {
>         /* Клієнт просить фабрику — не конструює сам.
>            Якщо логіка створення зміниться, правити лише ProductFactory. */
>         Product product = ProductFactory.createProduct("Мобільний телефон", 1500.0);
> 
>         System.out.println("Назва товару: " + product.getName());
>         System.out.println("Ціна товару: " + product.getPrice() + " грн");
> 
>         /* Результат:
>            Назва товару: Мобільний телефон
>            Ціна товару: 1500.0 грн */
>     }
> }
> ```

## 3. Controller (Контролер)

Покладай відповідальність за **обробку вхідних подій** та керування взаємодією між користувачем і системою на клас, що має найбільше інформації про потік управління.

> [!example] Практичне застосування
> 
> Клас `UserController` — це **контролер**, що приймає ім'я від користувача і делегує створення `User` до моделі. Він розділяє логіку введення даних і логіку їх обробки.
> 
> ```java
> import java.util.Scanner;
> 
> // Модель користувача — зберігає дані
> class User {
>     private String name;
> 
>     /* Проста модель: лише ім'я та метод доступу до нього */
>     public User(String name) {
>         this.name = name;
>     }
> 
>     public String getName() {
>         return name;
>     }
> }
> 
> // Контролер — обробляє запити та делегує роботу моделі
> class UserController {
>     /* createUser — єдина точка входу для створення User.
>        Контролер не знає нічого про введення з консолі —
>        він отримує вже підготовлені дані (name) і передає їх моделі. */
>     public User createUser(String name) {
>         return new User(name);
>     }
> }
> 
> public class Main {
>     public static void main(String[] args) {
>         UserController controller = new UserController();
>         Scanner scanner = new Scanner(System.in);
> 
>         /* Main координує: зчитує введення від користувача,
>            передає контролеру, отримує результат. */
>         System.out.print("Введіть ім'я користувача: ");
>         String userName = scanner.nextLine();
> 
>         User user = controller.createUser(userName);
> 
>         System.out.println("Створений користувач: " + user.getName());
> 
>         /* Результат (при введенні "Іван"):
>            Створений користувач: Іван */
>     }
> }
> ```

---

> [!warning] Обмеження та ризики
> 
> - **Суб'єктивність Expert.** Визначити «найкомпетентніший» клас не завжди очевидно — різні розробники можуть дійти різних висновків.
>     
> - **Creator ≠ завжди фабрика.** Принцип лише каже _хто_ має створювати, але не диктує патерн реалізації. Factory Method, Abstract Factory, Builder — все це різні способи втілення Creator.
>     
> - **Controller — не «Бог-клас».** Часта помилка — звалювати в контролер всю логіку. Контролер лише _координує_, а бізнес-логіка має жити в моделях або сервісах.
>     
> - **GRASP — не срібна куля.** Деякі архітектурні задачі виходять за межі цих принципів і потребують доповнення через GoF-патерни або архітектурні стилі (MVC, Hexagonal Architecture тощо).
>
