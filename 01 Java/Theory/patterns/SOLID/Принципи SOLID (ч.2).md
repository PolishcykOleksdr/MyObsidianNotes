> **SOLID (ч.2)** завершує розгляд принципів трьома останніми: **LSP**, **ISP** та **DIP** — усі три спрямовані на правильну організацію ієрархій, інтерфейсів і залежностей між компонентами системи.

---

> [!abstract] Core Framework 
> **Три принципи другої частини SOLID:**
> 
> - **L — LSP (Liskov Substitution Principle)** — підклас повинен повністю замінювати базовий клас без зміни очікуваної поведінки програми;
> - **I — ISP (Interface Segregation Principle)** — клієнти не повинні залежати від методів інтерфейсу, які вони не використовують; великі інтерфейси розбиваються на менші;
> - **D — DIP (Dependency Inversion Principle)** — модулі високого рівня залежать від абстракцій, а не від конкретних реалізацій; і ті, й інші залежать від абстракцій.

---

> [!info] Key Insights
> 
> - **LSP — контракт поведінки, не лише типу.** Підклас може замінити базовий клас тоді, коли він виконує той самий контракт: повертає очікуваний результат, не кидає несподіваних винятків і не порушує інваріанти базового класу.
>     
> - **ISP — «товсті» інтерфейси є запахом коду.** Якщо клас реалізує інтерфейс, але залишає деякі методи порожніми або з `UnsupportedOperationException` — це явне порушення ISP. Інтерфейс потрібно розбити.
>     
> - **DIP — «залежати від абстракцій» означає Dependency Injection.** Конкретна реалізація передається ззовні (через конструктор або сеттер), а не створюється всередині класу через `new`. Це і є інверсія: клас більше не контролює, що саме він отримає.
>     
> - **LSP і поліморфізм — нерозривні.** Поліморфізм працює коректно лише тоді, коли LSP дотримано. Порушення LSP призводить до того, що `instanceof`-перевірки починають «просочуватись» у код — сигнал проблеми.
>     
> - **DIP і тестованість.** Залежність від абстракції дозволяє у тестах підмінити реальну реалізацію (`EmailService`) на мок або стаб, не змінюючи жодного рядка у класі `Notification`.
>

## L — Liskov Substitution Principle (LSP)

Об'єкти підкласів повинні **замінювати об'єкти базового класу** без зміни очікуваної поведінки програми. Будь-де, де очікується `Shape`, має коректно працювати і `Rectangle`, і `Square`.

**Коли застосовувати LSP:**

- При наслідуванні — перевір, що підклас не порушує поведінку базового
- При роботі з інтерфейсами — реалізатор повинен повністю виконувати контракт
- При використанні поліморфізму — виклик методу через базовий тип повинен давати очікуваний результат

> [!example] Практичне застосування
> 
> `Rectangle` і `Square` обидва успадковують `Shape` і коректно реалізують `calculateArea()`. Їх можна безпроблемно використовувати через посилання типу `Shape` — LSP дотримано.
> 
> ```java
> /* Базовий клас для геометричних фігур.
>    Метод calculateArea() повертає 0 за замовчуванням —
>    підкласи зобов'язані перевизначити його. */
> class Shape {
>     public double calculateArea() {
>         return 0; /* Повертає нульову площу за замовчуванням */
>     }
> }
> 
> /* Клас для прямокутників.
>    Замінює Shape без порушення очікуваної поведінки. */
> class Rectangle extends Shape {
>     private double width;
>     private double height;
> 
>     public Rectangle(double width, double height) {
>         this.width = width;
>         this.height = height;
>     }
> 
>     /* Площа прямокутника: S = width * height */
>     @Override
>     public double calculateArea() {
>         return width * height;
>     }
> }
> 
> /* Клас для квадратів.
>    Квадрат — окремий підклас Shape, а не підклас Rectangle.
>    Це уникає класичної пастки LSP (Square extends Rectangle),
>    де setWidth() порушував би інваріант квадрата. */
> class Square extends Shape {
>     private double side;
> 
>     public Square(double side) {
>         this.side = side;
>     }
> 
>     /* Площа квадрата: S = side² */
>     @Override
>     public double calculateArea() {
>         return side * side;
>     }
> }
> 
> public class Main {
>     public static void main(String[] args) {
>         /* Посилання типу Shape — підкласи підставляються без жодних змін у коді */
>         Shape rectangle = new Rectangle(5, 4);
>         Shape square = new Square(5);
> 
>         /* JVM через dynamic dispatch викликає правильний calculateArea() */
>         System.out.println("Площа прямокутника: " + rectangle.calculateArea()); /* Очікується 20.0 */
>         System.out.println("Площа квадрата: " + square.calculateArea());        /* Очікується 25.0 */
>     }
> }
> ```
> 
> Формули площ:
> 
> - Прямокутник: $S = w \times h$
> - Квадрат: $S = a^2$

## I — Interface Segregation Principle (ISP)

Клієнти **не повинні залежати від інтерфейсів, які вони не використовують**. Великі «товсті» інтерфейси розбиваються на менші та специфічні — клас реалізує лише те, що йому потрібно.

**Коли застосовувати ISP:**

- При проектуванні нових інтерфейсів — одразу роби їх вузькоспеціалізованими
- Якщо клас реалізує інтерфейс, але деякі методи залишає порожніми — сигнал розбити інтерфейс
- При роботі з бібліотеками — залежати лише від тих методів, які реально використовуються

> [!example] Практичне застосування
> 
> `SimpleMultimediaPlayer` реалізує `MultimediaPlayer` і `AudioPlayer`, але **не** `VideoPlayer` — він не відтворює відео. ISP дозволяє класу залежати лише від потрібних контрактів.
> 
> ```java
> /* Інтерфейс для відтворення мультимедійного контенту.
>    Базові операції керування відтворенням. */
> interface MultimediaPlayer {
>     void play();  /* Відтворення контенту */
>     void pause(); /* Пауза */
>     void stop();  /* Зупинка */
> }
> 
> /* Окремий інтерфейс для перегляду відео.
>    Розділений від MultimediaPlayer — не кожен плеєр відтворює відео. */
> interface VideoPlayer {
>     void playVideo(); /* Відтворення відео */
> }
> 
> /* Окремий інтерфейс для програвання аудіо.
>    Розділений від MultimediaPlayer — не кожен плеєр відтворює аудіо. */
> interface AudioPlayer {
>     void playAudio(); /* Відтворення аудіо */
> }
> 
> /* Клас мультимедійного програвача.
>    Реалізує MultimediaPlayer і AudioPlayer, але НЕ VideoPlayer —
>    він не зобов'язаний реалізовувати те, що йому не потрібно.
>    Це і є суть ISP: клас залежить лише від потрібних інтерфейсів. */
> class SimpleMultimediaPlayer implements MultimediaPlayer, AudioPlayer {
>     @Override
>     public void play() {
>         System.out.println("Мультимедійний контент відтворюється");
>     }
> 
>     @Override
>     public void pause() {
>         System.out.println("Мультимедійний контент на паузі");
>     }
> 
>     @Override
>     public void stop() {
>         System.out.println("Мультимедійний контент зупинено");
>     }
> 
>     @Override
>     public void playAudio() {
>         System.out.println("Аудіо відтворюється");
>     }
> }
> 
> public class Main {
>     public static void main(String[] args) {
>         SimpleMultimediaPlayer player = new SimpleMultimediaPlayer();
>         player.play();      /* Очікується відтворення мультимедійного контенту */
>         player.playAudio(); /* Очікується відтворення аудіо */
>     }
> }
> ```

## D — Dependency Inversion Principle (DIP)

Модулі **високого рівня не залежать від модулів низького рівня** — обидва залежать від абстракцій. Класи залежать від інтерфейсів, а конкретні реалізації передаються ззовні.

**Коли застосовувати DIP:**

- При проектуванні систем з великою кількістю залежностей між компонентами
- При використанні інтерфейсів — клас верхнього рівня спирається на абстракцію, не на `new ConcreteClass()`
- При написанні тестів — легка підміна реалізації на мок без зміни коду класу

> [!example] Практичне застосування
> 
> `Notification` залежить від абстракції `MessageService`, а не від `EmailService` напряму. Підміна `EmailService` на будь-який інший сервіс (SMS, Push) — лише одна зміна в `Main`, без торкання `Notification`.
> 
> ```java
> /* Абстракція — інтерфейс сервісу відправки повідомлень.
>    Notification залежить від цього контракту, а не від реалізації. */
> interface MessageService {
>     void sendMessage(String message);
> }
> 
> /* Конкретна реалізація — відправка по електронній пошті.
>    Модуль низького рівня: знає деталі відправки email. */
> class EmailService implements MessageService {
>     @Override
>     public void sendMessage(String message) {
>         System.out.println("Відправлено електронного листа: " + message);
>     }
> }
> 
> /* Клас верхнього рівня — використовує сервіс через абстракцію.
>    Не знає і не хоче знати, як саме відправляється повідомлення.
>    Це і є інверсія залежностей: раніше Notification сам створював
>    EmailService (залежність від деталі), тепер отримує абстракцію ззовні. */
> class Notification {
>     /* final — після ін'єкції сервіс не підміняється */
>     private final MessageService messageService;
> 
>     /* Конструктор приймає абстракцію — Dependency Injection через конструктор.
>        Конкретна реалізація вирішується в точці складання (Main). */
>     public Notification(MessageService messageService) {
>         this.messageService = messageService;
>     }
> 
>     public void sendNotification(String message) {
>         /* Notification делегує відправку — не знає деталей реалізації */
>         messageService.sendMessage(message);
>     }
> }
> 
> public class Main {
>     public static void main(String[] args) {
>         /* Точка складання: тут вирішується, яка реалізація MessageService використовується.
>            Щоб перейти на SMS або Push — змінити лише цей рядок. */
>         MessageService emailService = new EmailService();
> 
>         /* Передача сервісу у Notification через конструктор */
>         Notification notification = new Notification(emailService);
> 
>         /* Відправка повідомлення через абстракцію */
>         notification.sendNotification("Привіт, це тестове повідомлення!");
> 
>         /* Результат:
>            Відправлено електронного листа: Привіт, це тестове повідомлення! */
>     }
> }
> ```

---

> [!warning] Обмеження та ризики
> 
> - **LSP — не лише про типи.** Компілятор перевіряє сумісність типів, але не поведінку. Порушення LSP часто непомітне на рівні синтаксису — воно проявляється у рантаймі у вигляді несподіваних результатів або винятків.
>     
> - **ISP і надмірне дроблення.** Розбивати інтерфейс на інтерфейс з одним методом — теж антипатерн. Мета ISP — змістовна гранулярність, а не мінімалізм заради мінімалізму.
>     
> - **DIP ≠ завжди інтерфейс.** Абстракцією може бути і абстрактний клас. Головне — щоб клас верхнього рівня не залежав від конкретної реалізації через `new`.
>     
> - **DIP збільшує кількість класів і конфігураційного коду.** У великих системах управління залежностями виноситься у DI-контейнери (Spring, Guice), інакше `Main` перетворюється на «клас-павутину» з сотнями `new`.
>     

---

> [!tip] Коли використовувати
> 
> **LSP** ✅:
> 
> - Будуєш ієрархію спадкування — перевіряй, що підклас виконує контракт базового
> - Використовуєш поліморфізм і очікуєш однакову поведінку незалежно від типу
> 
> **ISP** ✅:
> 
> - Інтерфейс має методи, які деякі реалізатори залишають порожніми
> - Різні клієнти потребують різних підмножин методів одного великого інтерфейсу
> 
> **DIP** ✅:
> 
> - Клас верхнього рівня безпосередньо створює об'єкти низького рівня через `new`
> - Потрібна легка підміна реалізації у тестах або у різних середовищах
> - Система має складну мережу залежностей між компонентами
