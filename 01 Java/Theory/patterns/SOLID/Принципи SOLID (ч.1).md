> **SOLID** — це набір основних принципів об'єктно-орієнтованого програмування, спрямованих на створення **гнучких**, **підтримуваних** і **розширюваних** програмних систем. Кожна літера абревіатури позначає окремий принцип.

---

> [!abstract] Core Framework 
> **Кожна літера SOLID — окремий принцип проектування:**
> 
> - **S** — Single Responsibility Principle (Одинична відповідальність)
> - **O** — Open/Closed Principle (Відкритість/Закритість)
> - **L** — Liskov Substitution Principle
> - **I** — Interface Segregation Principle
> - **D** — Dependency Inversion Principle
> 
> Розглянуті у цій частині: **S** та **O**.

> [!info] Key Insights
> 
> - **SRP — одна причина для зміни.** Якщо клас змінюється через дві різні причини (наприклад, зміна логіки збереження І зміна логіки відображення) — він порушує SRP і потребує розділення.
>     
> - **OCP — розширення без модифікації.** Новий функціонал додається через нові класи (підкласи, реалізації інтерфейсів), а не через редагування вже існуючого і перевіреного коду.
>     
> - **Поліморфізм — інструмент OCP.** Виклик абстрактного методу через батьківський тип дозволяє `AreaCalculator` обчислювати площу будь-якої нової фігури без жодної зміни у своєму коді.
>     
> - **SRP і High Cohesion — одна ідея.** SRP з SOLID і High Cohesion з GRASP описують те саме з різних кутів: клас повинен бути зосереджений на одній чіткій відповідальності.
>     
> - **OCP і Open/Closed — про стабільність існуючого коду.** Вже написаний і протестований код не торкається при додаванні нового функціоналу — ризик регресії зводиться до мінімуму.
>     

---

> [!tip] Переваги / Недоліки
> 
> **Так** ✅:
> 
> - Гнучкість і розширюваність — функціонал додається без зміни наявного коду
> - Покращена читабельність — менші, спеціалізовані класи простіші для розуміння
> - Підвищена стійкість до змін — менший ризик регресій
> - Сприяє повторному використанню компонентів
> - Покращує тестованість — кожен клас тестується ізольовано
> 
> **Ні** ❌:
> 
> - Збільшення кількості класів та інтерфейсів — вища початкова складність
> - Надмірна абстракція може сповільнити розробку
> - Вимагає додаткового навчання для команди
> - У маленьких проектах повне дотримання SOLID може бути невиправданим

## S — Single Responsibility Principle (SRP)

Клас повинен мати **лише одну причину для зміни**. Кожен клас відповідає лише за одну конкретну функціональність або поведінку.

**Коли застосовувати SRP:**

- При проектуванні нових класів — розділяй функціональність одразу
- Якщо клас має надмірну кількість методів різного призначення
- Якщо зміна однієї функції вимагає правки в багатьох місцях класу
- Під час рефакторингу — перевір, чи не перевантажений клас зайвою відповідальністю

> [!example] Практичне застосування
> 
> Клас `UserDataStorageAndDisplay` порушує SRP — він відповідає і за збереження, і за відображення. Після перепроектування кожна відповідальність отримує власний клас.
> 
> ```java
> /* ПОРУШЕННЯ SRP:
>    Клас відповідає за дві різні речі —
>    збереження у БД І відображення у консолі.
>    Зміна формату відображення або логіки БД
>    змушує чіпати один і той самий клас. */
> class UserDataStorageAndDisplay {
>     private List<User> users;
> 
>     public void saveUser(User user) {
>         /* Логіка збереження користувача в базі даних */
>     }
> 
>     public void displayUsers() {
>         /* Логіка відображення списку користувачів у консолі */
>     }
> }
> 
> /* ВИПРАВЛЕННЯ — клас лише для збереження.
>    Єдина причина для зміни: зміна логіки роботи з БД. */
> class UserDataStorage {
>     private List<User> users;
> 
>     public void saveUser(User user) {
>         /* Логіка збереження користувача в базі даних */
>     }
> }
> 
> /* ВИПРАВЛЕННЯ — клас лише для відображення.
>    Єдина причина для зміни: зміна формату виведення. */
> class UserDisplay {
>     private List<User> users;
> 
>     public void displayUsers() {
>         /* Логіка відображення списку користувачів у консолі */
>     }
> }
> ```

## O — Open/Closed Principle (OCP)

Система повинна бути **відкритою для розширення**, але **закритою для модифікації**. Нові можливості додаються через нові класи, а не через зміну вже наявного коду.

**Коли застосовувати OCP:**

- При проектуванні класів з потенційно змінною поведінкою
- При використанні поліморфізму — нові підкласи розширюють, не змінюючи базовий код
- При застосуванні патернів Стратегія, Декоратор, Фабрика — всі вони реалізують OCP

> [!example] Практичне застосування
> 
> `AreaCalculator` обчислює площу для масиву `Shape[]`. Додавання нового типу фігури (наприклад, `Triangle`) не потребує жодних змін у `AreaCalculator` — достатньо створити новий підклас `Shape`.
> 
> ```java
> /* Абстрактний клас — «точка розширення».
>    Новий тип фігури = новий підклас Shape.
>    AreaCalculator при цьому не змінюється. */
> abstract class Shape {
>     abstract double area();
> }
> 
> /* Конкретний клас прямокутника */
> class Rectangle extends Shape {
>     private double width;
>     private double height;
> 
>     public Rectangle(double width, double height) {
>         this.width = width;
>         this.height = height;
>     }
> 
>     /* Площа прямокутника: S = width * height */
>     @Override
>     double area() {
>         return width * height;
>     }
> }
> 
> /* Конкретний клас кола */
> class Circle extends Shape {
>     private double radius;
> 
>     public Circle(double radius) {
>         this.radius = radius;
>     }
> 
>     /* Площа кола: S = π * r² */
>     @Override
>     double area() {
>         return Math.PI * radius * radius;
>     }
> }
> 
> /* Клас, який обчислює загальну площу фігур.
>    ЗАКРИТИЙ для модифікації: код не зміниться при додаванні нових фігур.
>    ВІДКРИТИЙ для розширення: підтримує будь-який новий підклас Shape
>    через поліморфний виклик shape.area(). */
> class AreaCalculator {
>     public double calculateTotalArea(Shape[] shapes) {
>         double totalArea = 0;
>         for (Shape shape : shapes) {
>             /* JVM через dynamic dispatch викликає area()
>                конкретного підкласу — Rectangle, Circle або будь-якого нового. */
>             totalArea += shape.area();
>         }
>         return totalArea;
>     }
> }
> 
> public class Main {
>     public static void main(String[] args) {
>         Shape[] shapes = {new Rectangle(5, 4), new Circle(3)};
>         AreaCalculator calculator = new AreaCalculator();
>         double totalArea = calculator.calculateTotalArea(shapes);
>         System.out.println("Загальна площа фігур: " + totalArea);
> 
>         /* Результат:
>            Загальна площа фігур: 48.26... (20.0 + 28.274...) */
>     }
> }
> ```
> 
> Формули площ:
> 
> - Прямокутник: $S = w \times h$
> - Коло: $S = \pi r^2$

---

> [!warning] Обмеження та ризики
> 
> - **SRP не означає «один метод».** Клас може мати кілька методів, якщо всі вони служать одній відповідальності. Занадто агресивне дроблення веде до надлишку мікрокласів.
>     
> - **OCP вимагає правильного передбачення точок розширення.** Неможливо захистити все від змін — важливо визначити, _що_ буде змінюватись, і саме там зробити абстракцію.
>     
> - **Надмірна ієрархія підкласів.** OCP через спадкування може породити глибокі ієрархії, які важко підтримувати. Перевага надається композиції та інтерфейсам перед глибоким спадкуванням.
>     
> - **Не завжди підходить для малих проектів.** У невеликих системах введення абстракцій заради SOLID може збільшити складність без реальної вигоди.
>
