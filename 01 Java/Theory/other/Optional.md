> Optional — об'єктний контейнер, що типізує наявність або відсутність значення. Він замінює небезпечні null-посилання декларативною обробкою результатів. Головна роль класу — служити типом повернення для методів, щоб явно попередити клієнта про можливу відсутність даних.

---

> [!abstract] Core Concept
> 
> **Головна ідея.** Формалізація відсутності значення через тип даних.
> 
> **Пояснення механіки.** Замість прямого доступу до об'єкта, клієнт взаємодіє з обгорткою, яка змушує обробити сценарій порожнечі.
> 
> **Коротка метафора.** Це конверт, який може бути або з листом, або порожнім; ви не можете прочитати лист, не відкривши конверт і не перевіривши його вміст.

---

> [!info] Key Insights
> 
> **Принцип 1. Явна сигналізація.** Використання Optional у сигнатурі методу зобов'язує розробника обробити випадок відсутності даних.
> 
> **Принцип 2. Декларативність.** Ланцюжки map та filter дозволяють будувати логіку трансформації даних без явних перевірок на null.
> 
> **Принцип 3. Тип повернення.** Призначений виключно для результатів методів, а не для збереження в полях класів.
> 
> **Принцип 4. Продуктивність.** Вибір між ледачим (orElseGet) та негайним (orElse) обчисленням дефолтних значень впливає на швидкість роботи.

---

> [!warning] Обмеження
> 
> Клас не реалізує Serializable, що робить його непридатним для полів у DTO або JPA-сутностях. Створює додаткове навантаження на Garbage Collector через алокацію об'єктів-обгорток.

---

> [!danger] Типові помилки
> 
> **Помилка 1. Виклик .get() без перевірки.** > Причина → Спроба отримати значення з порожнього контейнера.
> 
> Наслідок → NoSuchElementException.
> 
> Рішення → Замінити на orElse, orElseGet або використовувати ifPresent.
> 
> **Помилка 2. Optional як параметр методу.** > Причина → Змушує клієнта створювати зайві обгортки.
> 
> Наслідок → Захаращення коду та зайва алокація пам'яті.
> 
> Рішення → Передавати звичайні посилання та перевіряти на null всередині методу.

---

> [!note] Основні методи
>  - **Optional.of(T value)**
>     
>     - Що робить: Створює контейнер із гарантованим значенням.
>         
>     - Нюанси та ризики: Викидає NullPointerException, якщо вхідний параметр null.
>         
>     - Приклад Java:
>         
>        
>         
>         ```java
>         /* Use only when value is guaranteed to exist */
>         Optional<String> opt = Optional.of("Hillel"); 
>         ```
>         
> - **Optional.ofNullable(T value)**
>     
>     - Що робить: Створює Optional, який буде порожнім, якщо передано null.
>         
>     - Нюанси та ризики: Найбільш безпечний метод для обгортання сторонніх даних.
>         
>     - Приклад Java:
>         
>         
>         
>         ```java
>         /* Safe wrap of potentially null value */
>         Optional<String> opt = Optional.ofNullable(externalService.getData()); 
>         ```
>         
> - **T orElse(T other)**
>     
>     - Що робить: Повертає дефолтне значення, якщо контейнер порожній.
>         
>     - Нюанси та ризики: Об'єкт `other` обчислюється завжди, навіть якщо значення є. Це може уповільнювати систему.
>         
>     - Приклад Java:
>         
>         Java
>         
>         ```java
>         /* Constant default value */
>         String result = opt.orElse("Default"); 
>         ```
>         
> - **T orElseGet(Supplier<\? extends T> other)**
>     
>     - Що робить: Повертає значення або викликає Supplier, якщо значення відсутнє.
>         
>     - Нюанси та ризики: Це "ледачий" метод. Викликається лише за реальної потреби. Рекомендовано для важких обчислень.
>         
>     - Приклад Java:
>         
>         Java
>         
>         ```java
>         /* Supplier called only if opt is empty */
>         String result = opt.orElseGet(() -> computeExpensiveDefault()); 
>         ```
>         
> - **Optional<U\> map(Function<\? super T, ? extends U> mapper)**
>     
>     - Що робить: Трансформує вміст за допомогою функції.
>         
>     - Нюанси та ризики: Якщо функція повертає null, результат буде порожнім Optional.
>         
>     - Приклад Java:
>         
>         Java
>         
>         ```java
>         /* Transform text to its length */
>         Optional<Integer> length = opt.map(String::length); 
>         ```
> - **Optional<U\> flatMap(Function<\? super T, Optional<\? extends U>> mapper)**
>     
>     - Що робить: Трансформує значення, де сама функція трансформації повертає `Optional`. Він «сплющує» (flattens) результат, щоб не отримати вкладеність виду `Optional<Optional<T>>`.
>         
>     - Нюанси та ризики: Якщо функція-мапер поверне `null` замість `Optional`, метод викине `NullPointerException`. Це головна відмінність від `map`, який толерантний до `null` (просто поверне порожній контейнер).
>         
>     - Приклад Java:
>         
>         Java
>         
>         ```java
>         /* Transform text to its length */
>         Optional<Address> good = user.flatMap(User::getAddress); 
>         ```
>        
>    
>

> [!info] Key Insights (Java 9+ Improvements)
> 
> **Метод `ifPresentOrElse(Consumer, Runnable)`** > Дозволяє обробити обидва сценарії (наявність і відсутність) в одному ланцюжку.
> 
> Java
> 
> ```java
> opt.ifPresentOrElse(
>     value -> System.out.println("Found: " + value),
>     () -> System.out.println("Not found")
> );
> ```
> 
> **Метод `or(Supplier<Optional<T>>)`** > Схожий на `orElseGet`, але повертає не значення, а інший `Optional`. Ідеально для ланцюжкових пошуків (Cache -> DB -> API).
> 
> Java
> 
> ```java
> Optional<User> user = cache.find(id)
>     .or(() -> database.find(id))
>     .or(() -> remoteApi.find(id));
> ```
> 
> **Метод `stream()`** > Перетворює `Optional` на `Stream` (0 або 1 елемент). Дозволяє елегантно фільтрувати списки.
> 
> Java
> 
> ```java
> List<User> activeUsers = userIds.stream()
>     .map(repository::findById) // повертає Optional
>     .flatMap(Optional::stream)  // витягує значення та ігнорує порожні
>     .collect(toList());
> ```

# Примітивні типи (OptionalInt/Long/Double)

Для роботи з примітивами існують спеціалізовані класи, щоб уникнути зайвого навантаження.

|**Тип**|**Проблема Optional<T>**|**Рішення**|
|---|---|---|
|**int**|`Optional<Integer>` робить Boxing/Unboxing|**OptionalInt**|
|**long**|Створює зайві об'єкти в пам'яті|**OptionalLong**|
|**double**|Споживає більше CPU на перетворення|**OptionalDouble**|

> [!note] Порівняння
>  
> **Optional<T\>** — для поодиноких об'єктів.
> **OptionalInt** — для примітивів (уникає автобоксингу).


---
> [!success] Best Practices (Золоті правила)
> 
> 1. **Не використовуй у колекціях.** Замість `List<Optional<User>>` повертай порожній `List<User>`. Колекція — це вже контейнер.
>     
> 2. **Ніколи не повертай null.** Якщо метод повертає `Optional`, замість `return null;` завжди використовуй `return Optional.empty();`.
>     
> 3. **Тільки для повернення.** Використовуй `Optional` переважно як тип повернення. Не передавай його в параметри методів і не зберігай у полях класів (він не `Serializable`).
>     
> 4. **Уникай Identity-метрик.** `Optional` — це _value-based_ клас. Ніколи не використовуй на ньому `==` (тільки `.equals()`), `synchronized` або `identityHashCode`.
