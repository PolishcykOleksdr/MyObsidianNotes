> Реалізації: TreeSet, HashSet, LinkedHashSet

> Уявіть, що **Set** — це магічний мішок з іграшками. Ви можете класти туди що завгодно, але як тільки ви намагаєтеся покласти іграшку, яка там вже є, вона просто зникає. В результаті у вас завжди лише унікальні речі, але залежно від типу "мішка", вони можуть лежати як попало або бути розставлені за зростом.

---
## 1. Концептуальний фундамент

> [!abstract] Core Framework
> **Головна ідея:** `Set` — це інтерфейс, що розширює `Collection`, який математично моделює множину, де кожен елемент $e$ є унікальним згідно з методом `e1.equals(e2)`.
> 
> **Механіка:**
> 
> - **Крок 1 (Хешування):** При додаванні об'єкта обчислюється його `hashCode()`.
>     
> - **Крок 2 (Перевірка):** Система шукає кошик (bucket). Якщо він не порожній, проводиться перевірка через `equals()`.
>     
> - **Крок 3 (Рішення):** Якщо дублікат знайдено, `add()` повертає `false`, стан пам'яті не змінюється.
>     
> 
> **Метафора:** Список гостей на закриту вечірку. Охоронець (Set) звіряє паспорт (equals/hashCode) і не пускає людину, якщо вона вже всередині.

---

## 2. Ключові принципи та логіка

> [!info] Key Insights
> **Принцип 1: Контракт hashCode/equals** — Для коректної роботи `HashSet`, якщо `e1.equals(e2)`, то їх хеш-коди обов'язково мають бути однаковими.
> 
> **Принцип 2: Впорядкованість vs Сортування** — `LinkedHashSet` зберігає порядок **вставки**, тоді як `TreeSet` забезпечує **природний порядок** (Natural Ordering).
> 
> **Логічний зв'язок:** `HashSet` забезпечує швидкість $O(1)$, а `TreeSet` додає функціонал навігації ($floor$, $ceiling$) ціною зниження швидкості до $O(\log n)$.

# Classes
### Set

> [!info] **Class TreeSet**
> `Class TreeSet` – набір сортованих об'єктів у вигляді дерева. Зберігає свої елементи у [червоно-чорному дереві](https://uk.wikipedia.org/wiki/%D0%A7%D0%B5%D1%80%D0%B2%D0%BE%D0%BD%D0%BE-%D1%87%D0%BE%D1%80%D0%BD%D0%B5_%D0%B4%D0%B5%D1%80%D0%B5%D0%B2%D0%BE), упорядковує свої елементи з урахуванням їх значень. Значно повільніше, ніж `HashSet`.
> ![[Red-black_tree_example.png]]

> [!info] **Class HashSet**
> `Class HashSet` – набір об'єктів або хеш-множинами, де кожен елемент має ключ – унікальний хеш-код. Зберігає свої елементи в [хеш-таблиці](https://en.wikipedia.org/wiki/Hash_table), є найефективнішою реалізацією, однак це не дає жодних гарантій щодо порядку ітерації.
![[hash_table.png]]

> [!info] **Class LinkedHashSet**
> `Class LinkedHashSet` - пов'язана хеш-множина. Реалізований у вигляді хеш-таблиці, через яку проходить пов'язаний список, упорядковує свої елементи в порядку, в якому вони були вставлені в комплект (порядок вставки).

## 3. Технічні характеристики

|**Реалізація**|**Complexity (Time)**|**Memory**|**Нюанс**|
|---|---|---|---|
|**HashSet**|$O(1)$|Medium|Найшвидший; використовує `HashMap` всередині.|
|**LinkedHashSet**|$O(1)$|High|Додаткові вказівники для збереження порядку ітерації.|
|**TreeSet**|$O(\log n)$|High|Червоно-чорне дерево; не дозволяє `null`.|
|**Set.of()**|$O(1)$|Low|Створює Immutable (незмінний) набір; Java 9+.|

## 4. Критичні межі та безпека

> [!warning] Limits & Constraints
> - **Місткість:** Теоретичний ліміт `HashSet` обмежений розміром масиву в `HashMap` ($2^{31}-1$), але практичний поріг залежить від `loadFactor` (за замовчуванням $0.75$).
>     
> - **TreeSet & Null:** Спроба додати `null` у `TreeSet` без спеціального компаратора призведе до негайного краху.
>     

>[!danger] Common Pitfalls
> **Mutable Elements в Set**
> 
> - **Причина:** Зміна полів об'єкта, які беруть участь у `hashCode()`, після того, як об'єкт додано в `Set`.
>     
> - **Наслідок:** Об'єкт стає "невидимим" для методу `contains()` або `remove()`, виникає витік пам'яті.
>     
> - **Рішення:** Використовуйте лише `Immutable` об'єкти як елементи `Set`.
>

## 5. Implementation Details

> [!note] Method Signature: public boolean add(E e)
> **Логіка:** Повертає `true`, якщо набір ще не містив зазначеного елемента. В `TreeSet` цей метод ініціює балансування дерева (Rotation).
> 
> **Ризики:** `ClassCastException` у `TreeSet`, якщо елемент не реалізує `Comparable`.

# Java

```java
/* Strict Java Implementation: Set Logic */
import java.util.*;

public class SetMastery {
    public static void main(String[] args) {
        // HashSet: Хаотичний порядок, максимальна швидкість
        Set<String> hashSet = new HashSet<>();
        hashSet.add("Java");
        
        // TreeSet: Автоматичне сортування (Natural Order)
        NavigableSet<Integer> treeSet = new TreeSet<>();
        treeSet.addAll(Arrays.asList(10, 5, 20, 15));
        
        /* Використання NavigableSet API */
        Integer ceilingVal = treeSet.ceiling(12); // Поверне 15 (найменше з тих, що >= 12)
        
        // Immutable Set: Спроба модифікації викличе UnsupportedOperationException
        Set<String> immutable = Set.of("Data", "Structure");
        
        /* * Під капотом HashSet.add():
         * map.put(e, PRESENT) == null
         */
    }
}
```

> [!note] Що і де краще використовувати?
> 
> 1. HashSet (Абсолютний лідер за швидкістю)
> 
> Це варіант "за замовчуванням". Він працює на основі хеш-таблиці.
> 
> - **Плюси:** Найшвидші операції додавання, видалення та пошуку завдяки хеш-функціям.
>     
> - **Коли обирати:** Коли вам просто потрібно зберігати унікальні елементи і абсолютно байдуже, у якому порядку вони там лежать.
>     
> - **Особливість:** Не гарантує жодного порядку. При ітерації елементи можуть "стрибати".
>     
> 
> 2. LinkedHashSet (Золота середина)
> 
> Це розширення `HashSet`, яке додатково підтримує зв'язаний список (LinkedList) через усі елементи.
> 
> - **Плюси:** Працює майже так само швидко, як `HashSet`, але **зберігає порядок вставки**.
>     
> - **Коли обирати:** Коли вам потрібна швидкість хеш-сету, але ви хочете, щоб при виводі елементи йшли в тому ж порядку, в якому ви їх додавали.
>     
> - **Мінус:** Споживає трохи більше пам'яті через підтримку зв'язків між вузлами.
>     
> 
> 3. TreeSet (Для сортування)
> 
> Ця структура базується на червоно-чорному дереві.
> 
> - **Плюси:** Елементи **завжди відсортовані** (або за алфавітом, або за вашою логікою). Має додаткові методи: `first()`, `last()`, `higher()`, `lower()`.
>     
> - **Коли обирати:** Коли вам критично важливо постійно мати відсортовані дані або потрібно шукати "найближчі" значення (діапазони).
>     
> - **Мінус:** Найповільніший серед сетів ($O(\log n)$ — це все ще швидко, але повільніше за "миттєвий" $O(1)$ у HashSet).

# Time Complexity: TreeSet, HashSet, LinkedHashSet

| **Операція**           | **HashSet**           | **LinkedHashSet**       | **TreeSet**                     |
| ---------------------- | --------------------- | ----------------------- | ------------------------------- |
| **add(E e)**           | $O(1)$                | $O(1)$                  | $O(\log n)$                     |
| **remove(Object o)**   | $O(1)$                | $O(1)$                  | $O(\log n)$                     |
| **contains(Object o)** | $O(1)$                | $O(1)$                  | $O(\log n)$                     |
| **Порядок елементів**  | Відсутній (хаотичний) | Порядок вставки         | Відсортований (за компаратором) |
| **Основа структури**   | Hash Table            | Hash Table + LinkedList | Red-Black Tree (Self-balancing) |
> [!quote] Що де краще?
> 
> **HashSet** (Максимальна швидкість)
> 
> **Найкраще підходить** для 90% задач, де потрібно просто зберігати унікальні об'єкти та миттєво їх перевіряти.
> 
> - **Чому**: Використовує хеш-таблицю. Пошук елемента не залежить від їх кількості.
>     
> - **Перевага**: Найнижчі витрати ресурсів на операції `add` та `contains`.
>     
> - **Операції**: `add`, `remove`, `contains` — все за $O(1)$.
>     
> 
> **LinkedHashSet** (Збереження порядку вставки)
> 
> **Найкраще підходить** для ситуацій, коли порядок елементів має значення (наприклад, список нещодавно переглянутих товарів).
> 
> - **Чому**: Це `HashSet`, який додатково підтримує двонаправлений зв'язаний список через усі елементи.
>     
> - **Перевага**: При ітерації (виводі на екран) елементи будуть саме в тому порядку, в якому ви їх додавали.
>     
> - **Операції**: Ті самі, що в `HashSet`, але ітерація трохи швидша.
>     
> 
> **TreeSet** (Сортування та діапазони)
> 
> **Найкраще підходить** для випадків, коли дані мають бути завжди впорядковані (список імен за алфавітом, ціни від меншої до більшої).
> 
> - **Чому**: Побудований на червоно-чорному дереві (самобалансоване дерево пошуку).
>     
> - **Перевага**: Надає методи для роботи з діапазонами: `subSet`, `headSet`, `tailSet`.
>     
> - **Операції**: `first()`, `last()`, `higher()`, `lower()`.


> [!important] Важливо
> **Важливо пам'ятати:** Для `HashSet` та `LinkedHashSet` критично важливо правильно перевизначати методи `hashCode()` та `equals()`. Для `TreeSet` об'єкти повинні реалізовувати інтерфейс `Comparable` або ви маєте передати `Comparator`.
