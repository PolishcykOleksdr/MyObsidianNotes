> Реалізації: LinkedList, ArrayDeque, PriorityQueue

> Уявіть чергу в магазині. Перший, хто підійшов, першим отримує товар (FIFO). `PriorityQueue` — це черга, де люди з «VIP-квитками» (вищим пріоритетом) проходять вперед, незалежно від того, коли вони прийшли.

---

### 1. Концептуальний фундамент

> [!abstract] Core Framework
> **Головна ідея**: `Queue` — це колекція, призначена для утримання елементів перед обробкою. Вона розширює `Collection`, додаючи специфічні методи для роботи з "головою" черги.
> 
> **Механіка**:
> 
> - **Крок 1**: Елемент додається в "хвіст" (tail) через `offer()` або `add()`.
>     
> - **Крок 2**: Система впорядковує елементи: `LinkedList` зберігає порядок вставки, а `PriorityQueue` перебудовує внутрішню **Binary Heap** (бінарну купу).
>     
> - **Крок 3**: Обробка завжди починається з "голови" (head) через `poll()` або `remove()`.
>     
> 
> **Метафора**: Конвеєр на заводі. Деталі кладуть з одного боку, а робот забирає їх з іншого в строгому порядку.

---

### 2. Ключові принципи та логіка

>[!info] Key Insights
> **Принцип 1: Failure Strategy** — Методи черги дублюються: одні викидають Exception (`add`, `remove`, `element`), інші повертають спеціальне значення `null`/`false` (`offer`, `poll`, `peek`).
> 
> **Принцип 2: Priority Ordering** — У `PriorityQueue` елементи не лежать відсортованими в пам'яті. Вони організовані як купа, де гарантується лише те, що "голова" — це мінімальний елемент.
> 
> **Логічний зв'язок**: Вибір реалізації залежить від алгоритму: `LinkedList` — для чесної черги, `PriorityQueue` — для планувальників задач.

---
### 3. Критичні межі та безпека

>[!warning] Limits & Constraints
> - **null-заборона**: `PriorityQueue` категорично забороняє `null` (викидає `NullPointerException`), оскільки не може порівняти `null` з іншими об'єктами.
>     
> - **Місткість**: `ArrayDeque` та `PriorityQueue` мають динамічний масив, який розширюється, що може викликати затримку при копіюванні даних.
>     

>[!danger] Common Pitfalls
> **ConcurrentModificationException**
> 
> - **Причина**: Спроба ітеруватися по черзі та видаляти елементи одночасно не через ітератор.
>     
> - **Наслідок**: Крах програми під час циклу.
>     
> - **Рішення**: Використовувати методи `poll()` у циклі `while(!queue.isEmpty())` або `ConcurrentLinkedQueue`.
>     

---

### 4. Implementation Details

>[!note] Method Signature: public E poll()
> **Логіка**: Повертає значення з голови черги та видаляє його. Якщо черга порожня, повертає `null` замість винятку. Це безпечний спосіб розвантаження черги.

# Java

```java
import java.util.*;

/* Дистиляція логіки черг */
public class QueueMastery {
    public static void main(String[] args) {
        // 1. PriorityQueue: Елементи впорядковуються за пріоритетом (природний порядок)
        Queue<Integer> pQueue = new PriorityQueue<>();
        pQueue.offer(42);
        pQueue.offer(10);
        pQueue.offer(100);

        /* Голова завжди буде 10, хоча вона додана другою */
        System.out.println("PQ Head: " + pQueue.peek()); // 10

        // 2. LinkedList: Працює як класична черга FIFO
        Queue<String> listQueue = new LinkedList<>();
        listQueue.offer("First");
        listQueue.offer("Second");

        /* Видалення голови */
        String processed = listQueue.poll(); // "First"
        
        // 3. Безпечна обробка
        while (!pQueue.isEmpty()) {
            /* Обробка в порядку пріоритету: 10, 42, 100 */
            System.out.println("Processing: " + pQueue.poll());
        }
    }
}
```

# Time Complexity:

| **Операція**           | **LinkedList**     | **PriorityQueue**   |
| ---------------------- | ------------------ | ------------------- |
| **offer() / add()**    | $O(1)$             | $O(\log n)$         |
| **poll() / remove()**  | $O(1)$             | $O(\log n)$         |
| **peek() / element()** | $O(1)$             | $O(1)$              |
| **contains(Object)**   | $O(n)$             | $O(n)$              |
| **Основа**             | Doubly Linked List | Binary Heap (Array) |

> [!quote] Що де краще?
> 
> **ArrayDeque** (Чистий FIFO та Стек)
> 
> **Найкраще підходить** для високопродуктивних черг та стеків, де не потрібне сортування.
> 
> - **Чому**: Працює на циклічному масиві. Не витрачає час на створення об'єктів-вузлів.
>     
> - **Перевага**: **Cache Locality**. Процесор швидше обробляє дані, що лежать підряд.
>     
> - **Операції**: `add`, `poll`, `peek`.
>     
> 
> **PriorityQueue** (Алгоритми та планувальники задач)
> 
> **Найкраще підходить** для задач, де елементи мають обслуговуватися згідно з їхньою важливістю.
> 
> - **Чому**: Використовує бінарну купу (Binary Heap). Найпріоритетніший елемент завжди "спливає" нагору за $\log n$.
>     
> - **Перевага**: Автоматичне сортування "на льоту".
>     
> - **Операції**: `offer`, `poll` (видаляє найменший/найбільший).
>     
> 
> #### **LinkedList** (Динамічні черги з видаленням усередині)
> 
> **Найкраще підходить** лише якщо черга є частиною ширшої логіки маніпуляції списком.
> 
> - **Чому**: Дозволяє миттєве видалення елемента, якщо ви вже маєте на нього посилання (Iterator).
>     
> - **Перевага**: Немає пауз на переаллокацію великих масивів.
>     
> - **Операції**: `add`, `poll`, `remove(Object)`.
