> Java Collections Framework – це:
> - Уніфікована архітектура для представлення колекцій та керування ними.
> - Набір інтерфейсів та класів, які їх реалізують, що встановлює певну ієрархію колекцій.

---

> [!abstract] **Інтерфейси** – абстрактні типи даних, які представляють колекції. Інтерфейси дають змогу маніпулювати колекціями незалежно від деталей.

---

> [!abstract] **Реалізації** – конкретні реалізації інтерфейсів колекції. По суті вони є структурами даних багаторазового використання.

---

> [!abstract] **Алгоритми** – методи, які виконують корисні обчислення, такі як пошук та сортування для об'єктів, які реалізують інтерфейси колекції. Алгоритми поліморфні: один і той самий метод може використовуватися в різних реалізаціях відповідного інтерфейсу колекції.

> [!info] **Переваги та недоліки**
> **Основні переваги Java Collections Framework:**
> 
> - Зменшує витрати на програмування, надаючи структури даних і алгоритми, щоб не писати їх самостійно.
> - Підвищує продуктивність, надаючи високопродуктивні реалізації структур даних і алгоритмів. Оскільки різні реалізації кожного інтерфейсу взаємозамінні, програми можна налаштовувати шляхом перемикання реалізацій.
> - Зменшує зусилля, необхідні для розробки, реалізації та вивчення API.
> - Сприяє повторному використанню програмного забезпечення, надаючи стандартний інтерфейс для колекцій і алгоритмів, за допомогою яких можна ними керувати.
> 
> **Основні недоліки Java Collections Framework:**
> 
> - Необхідність приведення до правильного типу.
> - Неможливість виконання перевірки типу під час компіляції.

# Interfaces

### Collection

> [!info] **Interface Collection**
> `Interface Collection` **(Колекція)** – корінь ієрархії колекції. Колекція є групою об'єктів, її елементів. 
> `Interface Collection` **реалізують усі колекції** та використовується для передачі колекцій та маніпулювання ними, коли потрібна максимальна універсальність. Деякі типи колекцій допускають дублювання елементів, інші - ні. Деякі впорядковані, інші ні. 
> `Interface Collection` **- основний інтерфейс**, який описує базові методи, які має будь-яка колекція.

---
### Set

> [!info] **Interface Set**
> `Interface Set` (Набір, Комплект) – колекція, яка не може містити елементи, що повторюються. 
> `Interface Set` **моделює** математичну абстракцію набору (бере правила з математики та змушує код працювати за ними) у даному випадку це створення списку, який автоматично відсікає дублікати, як це робить справжня математична множина. 
> `Interface Set` як для прикладу може використовуватися для **унікальних контактів** у телефоні, де один номер не може належати двом різним іменам.

> [!info] **Interface SortedSet**
> `Interface SortedSet` – набір, який підтримує свої елементи у порядку зростання. 
> Декілька додаткових операцій надаються, щоб скористатися перевагами упорядкування/сортування. 
> Сортовані набори використовуються для природньо впорядкованих наборів, таких як списки слів.

> [!info] **Interface NavigableSet**
> `Interface NavigableSet` - набір для навігації в Java Collection Framework. 
> - Успадковується від `SortedSet`. 
> - Поводиться як `SortedSet`, за винятком того, що є доступні методи навігації на додаток до механізмів сортування `SortedSet`. 
> - `NavigableSet` може переміщатися по набору у зворотньому порядку порівняно з порядком, визначеним у `SortedSet`. 
> - Доступ до набору `NavigableSet` та переміщення по ньому можна виконувати як у зростаючому, так і у спадному порядку.

---
### List

> [!info] **Interface List**
> `Interface List` (Список) – впорядкована колекція (іноді називається послідовністю). Списки можуть містити елементи, що повторюються.
> 
> Користувач `List` зазвичай має точний контроль над тим, де в списку кожен елемент вставлений, і може отримати доступ до елементів за цілочисельним індексом (позиції).

---
### Queue

> [!info] **Interface Queue**
> `Interface Queue` (Черга) – колекція, яка використовується для зберігання кількох елементів перед обробкою (в буфері). Крім базових операцій збору, черга забезпечує додаткові операції вставки, вилучення та перевірки.

**Черги** зазвичай, але не обов'язково, **упорядковують** елементи в порядку **FIFO** (first-in-first-out, "першим увійшов - першим вийшов").

> [!warning] Коли List а коли Set
> `List` доцільно, коли передбачається ситуація з однаковими елементами в колекції або вкрай **важливий порядок елементів** у колекції, також треба мати можливість звернутися **до елемента певної позиції**.
> 
> `Set` доцільно, коли **потрібна унікальність**, швидкий пошук елемента всередині колекції та порядок не важливий.
