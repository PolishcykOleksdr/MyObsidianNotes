Навіть якщо ви прибрали сеттери, сторонній код може змінити ваш об'єкт, якщо ви дасте йому прямий доступ до внутрішнього списку. **Defensive copy** — це створення дубліката даних, щоб ніхто не зміг зіпсувати оригінал.

---

> [!abstract] Core Framework **Defensive Copy** — це техніка копіювання змінних об'єктів під час їх отримання або повернення, щоб забезпечити цілісність внутрішнього стану класу.
> 
> **Візуальна метафора:** Це як **фотографія документа**. Ви даєте установі фото (копію), щоб вони могли робити на ній свої позначки. Ваш оригінальний документ (внутрішній стан) залишається у вас у безпеці.

---

> [!info] Key Insights
> 
> - **Вхідний захист.** У конструкторі ми копіюємо дані, щоб зовнішній код не міг змінити їх пізніше через своє посилання.
>     
> - **Вихідний захист.** У геттері ми повертаємо копію, щоб той, хто викликав метод, не зміг видалити або змінити елементи нашого внутрішнього списку.
>     
> - **Справжня імутабельність.** Без захисного копіювання клас із полями-колекціями (як `List`) ніколи не буде по-справжньому незмінним.
>     
> - **Витрати.** Створення копій споживає пам'ять, тому для величезних масивів даних шукають компроміси (наприклад, `Collections.unmodifiableList`).
>     

---

> [!example] Код у стилі "Захист"
> 
> Java
> 
> ```java
> public final class SecureVault {
>     private final List<String> secrets;
>     
> 	public SecureVault(List<String> inputSecrets) {
> 	    // Копіюємо на вході, щоб оригінальний список не впливав на нас
> 	    this.secrets = new ArrayList<>(inputSecrets);
> 	}
> 	
> 	public List<String> getSecrets() {
> 	    // Повертаємо нову копію, щоб ніхто не зміг очистити наш список
> 	    return new ArrayList<>(this.secrets);
> 	}
> }
> ```
