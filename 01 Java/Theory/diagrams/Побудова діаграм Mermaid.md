> Мермейд (Mermaid) — це JavaScript-інструмент, який використовує текстову розмітку для генерації діаграм та графіків у реальному часі. В Obsidian це дозволяє зберігати візуалізації безпосередньо у Markdown-файлах, забезпечуючи їхню версійність та легке редагування.

---

> [!abstract] Core Framework 
> Mermaid базується на декларативному підході до візуалізації даних. Основні типи структур, що підтримуються:
> 
> - **Flowcharts (Блок-схеми):** для опису алгоритмів та процесів.
>     
> - **Sequence Diagrams (Діаграми послідовності):** для візуалізації взаємодії об'єктів у часі.
>     
> - **Class Diagrams (Діаграми класів):** для відображення структури об'єктно-орієнтованих систем.
>     
> - **State Diagrams (Діаграми станів):** для опису логіки переходів системи.
>     

---

> [!info] Key Insights
> 
> - **Текст як код.** Діаграма не є статичним зображенням (PNG/JPG), а зберігається як звичайний текст у блоці коду `mermaid`.
>     
> - **Рендеринг на льоту.** Obsidian використовує вбудований парсер Mermaid для перетворення тексту в SVG при переході в режим перегляду (Reading Mode) або Live Preview.
>     
> - **Стилізація.** Можливість кастомізації вузлів та зв'язків за допомогою класів CSS або внутрішніх директив `style`.
>     
> - **Напрямок потоку.** Гнучке керування орієнтацією (TB — зверху вниз, LR — зліва направо).
>     

---

> [!example] Практичне застосування Створення базової блок-схеми:
> 
> Фрагмент коду
> 
> ```
> graph TD
>     A[Початок] --> B{Чи є Java?}
>     B -- Так --> C[JVM завантажує байт-код]
>     B -- Ні --> D[Встановити JDK]
>     C --> E((Кінець))
> ```
> 
> Створення діаграми класів для Java:
> 
> Фрагмент коду
> 
> ```
> classDiagram
>     class User {
>         -String name
>         -int age
>         +getName() String
>     }
>     class Admin {
>         -List~String~ permissions
>         +deleteUser(User u) void
>     }
>     User <|-- Admin : успадкування
> ```

---

> [!warning] Обмеження та ризики
> 
> - **Складність синтаксису.** При великій кількості вузлів та перехресних зв'язків код стає важким для читання ("spaghetti code").
>     
> - **Проблеми з рендерингом.** Занадто великі діаграми можуть виходити за межі контейнера нотатки або сповільнювати роботу інтерфейсу.
>     
> - **Спеціальні символи.** Використання дужок або лапок у назвах вузлів потребує екранування або використання синтаксису `node["Text (with brackets)"]`.
>     

---

> [!tip] Коли використовувати Так ✅:
> 
> - Документування архітектури системи або логіки методів.
>     
> - Візуалізація життєвого циклу об'єктів.
>     
> - Створення швидких прототипів бізнес-процесів.
>     
> 
> Ні ❌:
> 
> - Потрібна піксельна точність розташування елементів.
>     
> - Діаграма містить тисячі елементів (краще використовувати спеціалізований софт).
>     
> - Потрібно вставити готові художні ілюстрації.
>
