> **Use Case діаграма** — це високорівневе представлення функціональних вимог системи з точки зору користувача. Вона візуалізує ролі (акторів) та їхню взаємодію із системою, не заглиблюючись у внутрішню реалізацію чи порядок виконання кроків.

---

> [!abstract] Core Framework 
> Діаграма базується на чотирьох ключових об'єктах:
> 
> - **Актор (Actor):** Роль користувача, зовнішньої системи або пристрою.
>     
> - **Варіант використання (Use Case):** Одиниця функціональності (що робить система).
>     
> - **Система (System Boundary):** Межі модельованого продукту (сайт, додаток, модуль).
>     
> - **Зв'язок (Relationship):** Взаємодія між акторами та прецедентами.
>

> [!info] Key Insights: Актори та Ролі
> 
> - **Первинні актори:** Ініціюють взаємодію (зазвичай зображуються зліва).
>     
> - **Вторинні актори:** Реагують на дії або надають сервіси (зображуються справа).
>     
> - **Актор vs Користувач:** Актор — це абстрактна роль (наприклад, "Студент"), а користувач — конкретна людина (Іван). Один користувач може виконувати кілька ролей одночасно.
>

> [!example] Практичне застосування: Типи зв'язків
> 
> ### 1. Асоціація (Association)
> 
> Прямий зв'язок між актором і функцією. У Java це відповідає виклику методу контролера користувачем.
> 
> ### 2. Розширення (<\<extend>>)
> 
> Додаткова функціональність, яка виконується лише за певної умови.
> 
> - **Приклад:** Прецедент "Показати помилку" розширює "Вхід у систему" лише при умові `[invalid password]`.
>     
> 
> ### 3. Включення (<\<include>>)
> 
> Обов'язкова частина базового прецеденту. Використовується для винесення спільного коду.
> 
> - **Приклад:** Прецеденти "Переказати кошти" та "Зняти готівку" обидва включають (<\<include>>) прецедент "Перевірити баланс".
>     
> 
> ### 4. Генералізація (Generalization)
> 
> Успадкування поведінки. Нащадок отримує всі властивості предка.
> 
> - **Приклад:** Актори "Менеджер" та "Директор" можуть успадковуватись від актора "Працівник".
>

> [!example] Моделювання логіки (Java-аналогія) Хоча діаграма описує "що", її можна спроектувати як набір сервісів:

```java
/* Базовий інтерфейс для оплати (Генералізація) */
interface PaymentStrategy {
    void pay(double amount);
}

/* Конкретні реалізації (Дочірні прецеденти) */
class CardPayment implements PaymentStrategy {
    public void pay(double amount) { /* ... */ }
}

public class BankingService {
    /* Базовий прецедент */
    public void processTransaction() {
        /* Включення (Include) - обов'язкова перевірка */
        checkAuth(); 
        
        /* Розширення (Extend) - виконується за умови */
        if (isBalanceLow()) {
            sendLowBalanceNotification(); 
        }
    }

    private void checkAuth() { /* Обов'язкова логіка */ }
    private void sendLowBalanceNotification() { /* Додаткова логіка */ }
}
```

---

> [!warning] Поширені помилки проектування
> 
> - **Погляд програміста:** Не зображуйте деталі реалізації (наприклад, "Підключення до БД"). Користувач бачить лише "Зберегти профіль".
>     
> - **Назви:** Прецеденти мають починатися з дієслова (Зробити, Оплатити, Отримати).
>     
> - **Ізольовані елементи:** Кожен актор має бути пов'язаний хоча б з одним прецедентом.
>     

---

> [!tip] Поради зі створення
> 
> - Використовуйте **Draw.io** або мобільний **Lucidchart** для швидких начерків.
>     
> - Розташовуйте прецеденти зверху донизу в логічному порядку (наприклад, від реєстрації до виходу).
>     
> - Використовуйте коментарі (Note) для опису умов розширення (Preconditions).
>
