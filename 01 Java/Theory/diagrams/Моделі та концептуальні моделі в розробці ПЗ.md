> У програмуванні та розробці програмного забезпечення термін **"модель"** використовується для опису об'єктів, явищ або процесів реального світу, які моделюються в програмному коді для подальшої обробки. Модель може бути використана для аналізу, візуалізації, прогнозування або взаємодії з об'єктами в програмі.

---

> [!abstract] Core Framework
>  Концептуальні моделі є фундаментальними абстракціями, що розділяються на кілька типів залежно від цілей проектування:
> 
> - **Domain Model:** Відображення бізнес-логіки та об'єктів предметної області.
>     
> - **Data Model:** Структура даних та зв'язків у базі даних.
>     
> - **Architecture Model:** Компоненти та взаємозв'язки системи.
>     
> - **Process/Interaction/Behavior Models:** Динаміка роботи, послідовність дій та стани системи.
>     

---

> [!info] Key Insights
> 
> - **Абстракція домену.** Модель предметної області дозволяє зафіксувати бізнес-правила у вигляді класів та інтерфейсів ще до початку написання технічної логіки.
>     
> - **ER-проектування.** Концептуальна модель БД фокусується на сутностях та атрибутах, ігноруючи особливості конкретної СУБД на етапі проектування.
>     
> - **Стани та події.** Модель поведінки (State Diagram) критично важлива для систем зі складною внутрішньою логікою, як-от автомати або системи обробки замовлень, де реакція залежить від поточного стану.
>     

---

> [!example] Практичне застосування: Domain Model Реалізація концептуальної моделі для системи управління продуктами:

```java
/* Клас, що представляє сутність "Продукт" */
public class Product {
    private long id;
    private String name;
    private double price;
    private Category category;

    /* Гетери та сетери */
}

/* Клас, що представляє сутність "Категорія продуктів" */
public class Category {
    private long id;
    private String name;
    private List<Product> products;

    /* Гетери та сетери */
}

/* Клас, що представляє сутність "Користувач" */
public class User {
    private long id;
    private String username;
    private String email;
    private List<Order> orders;

    /* Гетери та сетери */
}

/* Клас, що представляє сутність "Замовлення" */
public class Order {
    private long id;
    private User user;
    private List<Product> products;

    /* Гетери та сетери */
}
```

---

> [!example] Практичне застосування: Data & interaction
> 
> ### 1. Концептуальна модель БД (Student Courses)
> 
> - **Course:** `id (PK)`, назва, викладач.
>     
> - **Student:** `id (PK)`, ім'я, прізвище, група.
>     
> - **Grade:** `id (PK)`, `студент_id (FK)`, `курс_id (FK)`.
>     
> 
> ### 2. Модель взаємодії (Interaction Model)
> 
> Визначає сценарії роботи між Користувачем, Системою, Складом, Банком та іншими компонентами. Наприклад, Система оновлює кошик після додавання товарів користувачем та ініціює запит до Банку для оплати.
> 
> ### 3. Модель поведінки (State Diagram)
> 
> Приклад кавового автомата:
> 
> - **State: Готовий до замовлення** -> (Подія: Вибір напою) -> **State: Приготування напою**.
>     
> - Після завершення приготування система повертається у вихідний стан.
>     

---

> [!warning] Обмеження та ризики
> 
> - **Складність підтримки.** Надмірно деталізовані концептуальні моделі можуть швидко застаріти, якщо документація не оновлюється разом із кодом.
>     
> - **Розрив між моделлю та реалізацією.** Важливо, щоб сутності в коді (Java-класи) чітко відповідали сутностям у концептуальній моделі бази даних для запобігання логічних помилок.
>     

---

> [!tip] Коли використовувати Так ✅:
> 
> - При проектуванні нової системи для узгодження логіки з бізнес-замовником.
>     
> - Для візуалізації складних багатоетапних процесів (наприклад, онлайн-покупка).
>     
> - При створенні архітектури мікросервісів для визначення меж відповідальності (Bounded Contexts).
>     
> 
> Ні ❌:
> 
> - Для дуже простих скриптів або утиліт, де структура даних очевидна.
>     
> - Коли модель стає занадто абстрактною і не допомагає у розумінні коду.
>
