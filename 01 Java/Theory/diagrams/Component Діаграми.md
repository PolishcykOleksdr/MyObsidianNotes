> Component-діаграма — це статична структурна діаграма UML, що описує організацію та зв’язки між набором логічних або фізичних компонентів системи. Вона фокусується на архітектурі системи, розділяючи її на незалежні змінні модулі (компоненти), які взаємодіють через визначені інтерфейси.

---

> [!abstract] Core Framework 
> Компонентна діаграма базується на принципах модульності та інкапсуляції, де основними елементами виступають:
> 
> - **Component (Компонент):** Автономна частина системи, яка інкапсулює свій вміст і маніфестує свою поведінку через інтерфейси.
>     
> - **Interface (Інтерфейс):** Визначає набір операцій, які компонент надає (Provided Interface) або яких він потребує (Required Interface).
>     
> - **Port (Порт):** Точка взаємодії між компонентом і його оточенням, що групує інтерфейси.
>     
> - **Connector (Конектор):** Зв'язок, що забезпечує передачу сигналів або викликів між компонентами.
>     

---

> [!info] Key Insights
> 
> - **Інкапсуляція логіки.** Компонент приховує внутрішню реалізацію (класи, об'єкти), виставляючи назовні лише "контракти" у вигляді інтерфейсів.
>     
> - **Замінність (Replaceability).** Будь-який компонент можна замінити іншим, якщо він реалізує ті ж самі інтерфейси, що робить систему гнучкою до змін.
>     
> - **Чорна скринька (Black Box).** На рівні діаграми компонентів ми не бачимо внутрішнього коду — ми бачимо лише входи та виходи.
>     
> - **Типи інтерфейсів.** "Lollipop" (кулька) — наданий інтерфейс; "Socket" (півколо) — необхідний інтерфейс.
>

> [!example] Практичне застосування 
> В контексті Java-розробки компонент може відповідати JAR-файлу, модулю Spring Boot або окремому сервісу.
> 
> Приклад структури компонента (концептуально):
> 
> ```java
> /*
>  * Компонент: OrderProcessingSystem
>  * Надає інтерфейс: OrderService
>  * Потребує інтерфейс: PaymentGateway
>  */
>  
> public interface OrderService { 
> 	/* Наданий інтерфейс (Provided Interface) */ 
> 	void placeOrder(String item, int quantity); 
> }
> public interface PaymentGateway {
> 	 /* Необхідний інтерфейс (Required Interface) */ 
> 	 boolean processPayment(double amount); 
> }
> public class OrderComponent implements OrderService { 
> 	private final PaymentGateway paymentGateway;
> 	 // Залежність (Required Interface)
> 	 
> 	public OrderComponent(PaymentGateway gateway) {
> 	    this.paymentGateway = gateway;
> 	}
> @Override
> public void placeOrder(String item, int quantity) {
>     /* Внутрішня реалізація компонента прихована від зовнішнього світу */
>     double price = calculatePrice(item, quantity);
>     
>     if (paymentGateway.processPayment(price)) {
>         System.out.println("Order processed successfully");
>     }
> }
> private double calculatePrice(String item, int quantity) {
>     return 100.0 * quantity; // Логіка всередині "Black Box"
> }
> 
> }
> ```

---

> [!warning] Обмеження та ризики
> 
> - **Рівень деталізації.** Занадто дрібні компоненти перетворюють діаграму на хаос з ліній, занадто великі — роблять її неінформативною.
>     
> - **Відрив від реальності.** Якщо діаграма не синхронізується з фактичною структурою пакетів/модулів у коді, вона швидко стає марною.
>     
> - **Складність зв'язків.** Циклічні залежності між компонентами вказують на погану архітектуру (High Coupling), що нівелює переваги модульності.
>     

---

> [!tip] Коли використовувати **Так ✅:**
> 
> - Потрібно візуалізувати високорівневу архітектуру системи.
>     
> - Необхідно показати сторонні бібліотеки або API, від яких залежить проект.
>     
> - При проектуванні мікросервісної архітектури.
>     
> 
> **Ні ❌:**
> 
> - Для опису внутрішньої логіки одного методу або класу.
>     
> - Якщо система складається з 2-3 класів (діаграма буде надлишковою).
>     
> - Для опису послідовності викликів у часі (для цього є Sequence Diagram).
>
