> **UML (Unified Modeling Language)** — це стандартна мова для візуалізації, специфікації та документування програмних систем. Вона надає графічний інструментарій для опису структури (статичної частини) та поведінки (динамічної частини) системи. UML-діаграми поділяються на **14 типів**.

---

> [!abstract] Core Framework
>  UML базується на наборі стандартних елементів та відношень:
> 
> - **Елементи:** Класи, об'єкти, інтерфейси, пакети.
>     
> - **Відношення:** Асоціація, агрегація, композиція, спадкування, реалізація, залежність.
>     
> - **Організація:** Пакети для структурування моделі та Профілі для власних нотацій.
>     
> - **Поведінкові аспекти:** Діаграми станів, діяльності та взаємодії.
>     

---

> [!info] Key Insights: 
> Activity vs Sequence 
> Часто виникає питання порівняння Діаграми діяльності та Діаграми послідовності:
> 
> - **Sequence Diagram:** Фокусується на часовому порядку виконання дій та обміні повідомленнями між об'єктами.
>     
> - **Activity Diagram:** Показує цілісну роботу всієї системи та перехід від однієї дії до іншої. Це розширений варіант блок-схеми, що підтримує паралелізм.
>     
> - **Паралелізм:** На відміну від звичайних блок-схем, UML Activity Diagram має стандартизовану нотацію для одночасного виконання дій та їх синхронізації.
>

> [!example] Практичне застосування: 
> Елементи діаграми діяльності 
> Потік керування на діаграмі може бути послідовним, розгалуженим або одночасним.
> 
> ### Основні вузли керування:
> 
> 1. **Вилка (Fork):** Розділяє один потік на кілька одночасних завдань.
>     
> 2. **Злиття (Join):** Об'єднує паралельні потоки в один.
>     
> 3. **Вузол прийняття рішень (Decision Node):** Вибирає один істинний потік (аналог `if` у Java). Умови записуються в `[квадратних дужках]`.
>     
> 
> ### Доріжки (Swimlanes):
> 
> Спосіб групування дій за акторами (виконавцями). Рекомендується використовувати не більше 5 доріжок для збереження читабельності.

> [!quote] Join Node vs Merge Node на діаграмі діяльності
> 
> Хоча вони виглядають схоже (лінії сходяться), їхня логіка протилежна:
> 
> - **Join Node** (Синхронізація): чекає, поки **всі** вхідні паралельні потоки завершаться, перш ніж рухатися далі.
>     
> - **Merge Node** (Злиття): працює як «АБО». Як тільки **будь-який один** із вхідних потоків приходить до нього, він одразу передає керування далі.

> [!example] Алгоритм створення (Java-логіка) Хоча діаграма візуальна, її логіка відповідає структурному програмуванню:

```java
/* Логіка вузла прийняття рішень на прикладі входу в систему */
if (isCredentialsValid()) {
    /* Перехід до дії "Доступ дозволено" */
    showDashboard();
} else {
    /* Перехід до дії "Показати помилку" */
    showErrorMessage();
}

/* Логіка паралельного виконання (Fork/Join) */
Thread task1 = new Thread(() -> doActionA());
Thread task2 = new Thread(() -> doActionB());
task1.start();
task2.start();
/* Очікування завершення обох потоків (синхронізація) */
task1.join();
task2.join();
```

> [!warning] Поради щодо проектування
> 
> - **Інструментарій:** У Draw.io необхідно увімкнути бібліотеки `UML 2.5` та `SysML` через меню "Більше обрисів".
>     
> - **Візуальна чіткість:** Використовуйте "горбочки-переходи" (line jumps) у меню стилів Draw.io, якщо лінії потоків перетинаються.
>     
> - **Доріжки:** Для створення доріжок шукайте фігури за назвою `pool`.
>     
> - **Альтернативи:** Для професійної роботи можна використовувати Lucidchart або Microsoft Visio з наступним імпортом у Draw.io.
>

> [!tip] Сценарії використання діаграм діяльності
> 
> - **Діаграма виплати зарплати:** Моделювання фінансових потоків.
>     
> - **Діаграма приготування напою:** Опис алгоритмів роботи автоматів.
>     
> - **Діаграма входу в систему:** Опис логіки аутентифікації.
>     
> - **Робота з документом:** Моделювання життєвого циклу файлів.
>
