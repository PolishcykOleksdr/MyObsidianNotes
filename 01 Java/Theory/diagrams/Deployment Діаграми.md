> Deployment-діаграма — це тип структурної діаграми UML, яка описує фізичне розгортання артефактів на вузлах (nodes). Вона візуалізує, як програмне забезпечення розподіляється по апаратному забезпеченню, мережеві конфігурації та топологію системи в реальному середовищі.

---

> [!abstract] Core Framework 
> Фізична архітектура системи базується на таких елементах:
> 
> - **Node (Вузол):** Обчислювальний ресурс (сервер, ПК, смартфон) або пристрій, на якому виконується ПЗ.
>     
> - **Artifact (Артефакт):** Фізичний результат розробки (JAR-файл, WAR, EXE, скрипт бази даних), що розміщується на вузлі.
>     
> - **Device (Пристрій):** Апаратний вузол без обчислювальних потужностей (маршрутизатор, модем).
>     
> - **Execution Environment (Середовище виконання):** ПЗ всередині вузла, яке дозволяє виконувати артефакт (JVM, Docker Container, Application Server).
>     
> - **Communication Path (Шлях зв'язку):** Мережеве з'єднання між вузлами (TCP/IP, HTTP, JDBC).
>     

---

> [!info] Key Insights
> 
> - **Фізичний рівень.** На відміну від Component-діаграми, яка є логічною, Deployment-діаграма показує, де саме "лежать" файли на диску або в хмарі.
>     
> - **Вкладеність вузлів.** Вузли можуть містити інші вузли або середовища виконання (наприклад, Physical Server -> OS -> Docker -> JVM).
>     
> - **Масштабованість.** Діаграма допомагає виявити вузькі місця в інфраструктурі та спланувати розподіл навантаження.
>     
> - **Зв'язок маніфестації.** Артефакт маніфестує (реалізує) один або декілька компонентів системи.
>

> [!example] Практичне застосування 
> Розглянемо розгортання типового Java Spring Boot додатка з базою даних PostgreSQL.

```java
/*
 * Логіка зв'язку між вузлами (Deployment Context)
 * Вузол 1: Application Server (Ubuntu + Docker + JRE)
 * Вузол 2: Database Server (PostgreSQL)
 */

public class InfrastructureConfig {
    /* * Опис шляху зв'язку (Communication Path) через JDBC 
     * Протокол: TCP/IP, Порт: 5432
     */
    private String dbUrl = "jdbc:postgresql://db-node-01:5432/school_db";
    
    /* * Артефакт: school-service.jar 
     * Розміщується (Deployed to) на JVM всередині Docker-контейнера
     */
    public void startService() {
        System.out.println("Artifact school-service.jar is running on Node: AppServer");
    }
}
```

---

> [!warning] Обмеження та ризики
> 
> - **Надмірна деталізація.** Не варто малювати кожен кабель; фокусуйтеся на логічних каналах зв'язку.
>     
> - **Динамічність середовища.** У сучасних хмарних рішеннях (K8s) вузли створюються та видаляються автоматично, що робить статичну діаграму розгортання швидко застарілою.
>     
> - **Конфіденційність.** Уникайте вказання реальних IP-адрес або паролів на діаграмах архітектури.
>     

---

> [!tip] Коли використовувати
>  **Так ✅:**
> 
> - Потрібно пояснити системним адміністраторам, як інсталювати систему.
>     
> - Необхідно задокументувати мережеві протоколи та порти.
>     
> - Опис структури кластерів або розподілених систем.
>     
> 
> **Ні ❌:**
> 
> - Для простих десктопних додатків без мережевої взаємодії.
>     
> - На етапі раннього проектування логіки (краще Class або Sequence Diagram).
>
