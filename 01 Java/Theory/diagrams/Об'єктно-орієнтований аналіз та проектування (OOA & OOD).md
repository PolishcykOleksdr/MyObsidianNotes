> **Об'єктно-орієнтований аналіз та проектування (ООА та ООП)** — це методологія розробки ПЗ, що базується на використанні діаграм для моделювання різних аспектів системи. Вона дозволяє розробникам та аналітикам структурувати функціональні вимоги, логічну архітектуру та фізичне розгортання продукту.

---

> [!abstract] Core Framework 
> ООА та ООП використовують набір UML-діаграм, які поділяються на структурні (статичні) та поведінкові (динамічні):
> 
> - **Структурні:** Class, Component, Deployment Diagrams.
>     
> - **Поведінкові:** Use Case, Sequence, Activity, State Diagrams.
>     

---

> [!info] Key Insights
> 
> - **Комплексне моделювання.** Жодна діаграма не дає повної картини. Наприклад, Class Diagram описує "що" є в системі, а Sequence Diagram — "як" ці елементи взаємодіють у часі.
>     
> - **Рівні абстракції.** Процес переходить від концептуальних вимог (Use Case) до технічної реалізації класів (Class) і врешті до фізичного розміщення на серверах (Deployment).
>     
> - **Взаємозв'язок компонентів.** Діаграма компонентів фокусується на модульності, що критично для підтримки та масштабування Java-застосунків.
>     

---

> [!example] Практичне застосування: Типи діаграм
> 
> ### 1. Функціональний аналіз
> 
> - **Use Case Diagrams:** Опис функціональності з точки зору акторів (користувачів). Показують, які задачі система має виконувати.
>     
> 
> ### 2. Проектування структури (Static View)
> 
> - **Class Diagrams:** Визначення атрибутів, методів та зв'язків (Inheritance, Association).
>     
> - **Component Diagrams:** Моделювання модульної структури та залежностей.
>     
> - **Deployment Diagrams:** Фізичне розташування компонентів на апаратному забезпеченні (вузлах).
>     
> 
> ### 3. Моделювання динаміки (Dynamic View)
> 
> - **Sequence Diagrams:** Послідовність викликів між об'єктами.
>     
> - **Activity Diagrams:** Послідовність дій та прийняття рішень (бізнес-процеси).
>     
> - **State Diagrams:** Поведінка об'єкта залежно від його поточного стану та вхідних подій.
>     

---

> [!example] Приклад реалізації зв'язку Class -> Sequence У Java-коді Class Diagram визначає структуру, а Sequence Diagram — потік виконання методів:

```java
/* Опис структури з Class Diagram */
public class OrderService {
    private InventoryManager inventory; /* Асоціація */

    public void processOrder(Order order) {
        /* Логіка, що візуалізується на Sequence Diagram */
        if (inventory.checkStock(order.getItemId())) {
            inventory.reserveItem(order.getItemId());
            System.out.println("Order processed");
        }
    }
}

public class InventoryManager {
    public boolean checkStock(long id) {
        /* Перевірка наявності */
        return true;
    }

    public void reserveItem(long id) {
        /* Резервування */
    }
}
```

---

> [!warning] Обмеження та ризики
> 
> - **Надмірність.** Створення всіх типів діаграм для маленького проекту призводить до "паралічу аналізу". Слід обирати лише ті, що додають цінність.
>     
> - **Невідповідність коду.** Найбільший ризик — коли діаграми існують окремо від живої кодової бази Java, стаючи неактуальними після першого рефакторингу.
>     

---

> [!tip] Коли використовувати Так ✅:
> 
> - **Complex Logic:** Для візуалізації складних взаємодій між десятками мікросервісів (Sequence/Component).
>     
> - **Stakeholder Meetings:** Use Case діаграми ідеальні для обговорення вимог із бізнесом.
>     
> - **Onboarding:** Для швидкого введення нових інженерів у проект.
>     
> 
> Ні ❌:
> 
> - Для документування тривіальних CRUD-операцій.
>     
> - Якщо команда практикує екстремальне програмування, де код є єдиним джерелом істини.
>
